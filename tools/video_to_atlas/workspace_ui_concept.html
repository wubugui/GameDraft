我所理解的产品形态：单工作区 + 持久化 + 导入弹窗

与当前代码的差异摘要： 当前 gui.py 为左右分栏（左侧纵向表单含视频/色键/帧库/动画/导出，右侧 QVideoWidget）。 按你的描述，主界面应不再以左右分栏组织，而是帧库与动画、导出工作台为主的单一滚动工作区；所有工程数据持久化（帧像素、帧元数据、动画序列及其序列侧属性、导出列表的「仅导出用」配置等，具体 schema 实现时再定）。 素材库结构：工作区内登记视频源，每视频一条节点，其下为子帧库。帧像素不对翻转/缩放；删帧、追加抽帧等在各子库内进行。 激活子库（接旧工作流）：任意时刻有且仅有一个被激活的视频子帧库（在树/列表中选中该视频即激活）。第 1～2 节的快捷键选首尾、蒙版对照、范围预览、智能匹配，以及「从首尾范围新建/扩展动画」等凡依赖「当前帧库」的操作，一律以激活子库为基准；未激活时禁用或提示先选视频。其它视频的帧仍存在工程中，但不在此编辑上下文中参与。 帧标识（id 机制）：改造后动画序列不得仅用「在某视频下的0..n序号」持久化引用帧。每一帧在写入工程时分配全局唯一且稳定的 frame_id（建议 UUID/十六进制串，与现有 FrameItem.id 思路一致）。增：新抽帧得到新 id。删：从子库移除该 id，工程中所有仍引用该 id 的动画槽位变为悬空，需 UI 提示并允许清理或替换（具体策略实现时定）。改：不就地改像素则 id 不变；若业务上「替换素材」可视为删旧加新并更新引用。界面上的「本地索引」仅便于在当前激活子库内浏览，排序变化不改变其它帧 id；动画列表里展示的是 frame_id（可附所属视频名与本地序号作注释）。 数据边界：动画序列存的是frame_id 的有序列表（加每槽位水平翻转标记等），序列不提供缩放；编辑序列不得改写任一子帧库像素。分辨率缩放仅属于导出工作台。 视频加载、预览、区间、色键、抽帧归入「导入工具」独立窗口；确认导入时在工作区创建或选中视频节点，抽得的帧写入该节点下子帧库。不提供从已导出图集/meta 再导入：工程内动画已持久化，需要调整时在工程里改序列并重新导出即可。

需求对齐确认（我理解的重点）
  • 产品定位：帧 + 动画数据管理，主界面是工作区而非「半边视频编辑器」。
  • 持久化：工作区文件（或目录）承载视频源列表、各视频子帧库像素与元数据、动画序列与导出列表配置；打开软件即加载。
  • 素材库与激活：多视频多子库；选中即激活，后续「帧库向」操作全部以激活子库为基准，承接原单库工作流。
  • frame_id：每帧稳定全局 id；动画序列只引用 id，不用子库本地序号持久化；增删帧时维护 id 与悬空引用策略。
  • 子库内容：仅存该视频抽得的原始素材；无子库内翻转与缩放。
  • 首尾工作区：保留蒙版对照；旁侧增加基于首尾的范围循环预览与调速。
  • 智能匹配：在限定范围内搜首/尾候选，预览结果，一键应用。
  • 动画库：frame_id 有序列表；可跨视频；从激活子库取范围时做「序号→id」转换；序列侧仅水平翻转（无缩放），不回写子库像素。
  • 导出：多选入列表；每行独立导出参数为导出专用（含分辨率缩放、边距、spritesheet、世界尺寸等），不改变动画序列里已保存的序列数据。
  • 视频路径：整体迁入导入弹窗，色键属于导入流程内步骤。
  • 无图集回灌：不实现从导出物 meta/PNG 再导入；动画序列随工作区持久化，修改后重新导出即可。
若某条与你想的不一致（例如智能匹配算法的输入范围或序列翻转标记的粒度），可再收窄一版线框。
