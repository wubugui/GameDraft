# 主编辑器系统性错误：为什么会犯 + 以后怎么不再犯

> 配套 artifact/Reviews/主编辑器全面审查-2026-07-14.md(2 P0 + 19 P1 + P2/P3 已全部修复验证通过)。
> 本轮所有严重问题都落在"444+59 测试全绿"之下——说明现有测试体系有结构性盲区。
> 本文把每一类系统性错误归因到"测试盖不到的那一层",并给出这次补的护栏 + 今后的硬规则。
> 「今后硬规则」四条 + 验收门升级已投 agent_docs/_meta/inbox/ 待治理正式收编进 editor-tools/norms.md。

## 一句话总结

**这次所有严重问题都不是"逻辑写错",而是"正确的逻辑接在了错误的层/漏接了一条路/护栏测在了错误的抽象层"。** 单元测试全绿掩盖了它们,因为测试大多在 model 层往返和表单探针上,而 bug 在流程层(门控/切换/撤销)、交互层(真实鼠标/键盘)、登记面(手工镜像清单)和缝合面(编辑器之间/编辑器与 Save All)。

---

## 六类系统性错误 × 根因 × 护栏 × 规则

### 根因1：护栏测在了比 bug 更低的抽象层(最致命)

**表现**：
- P0-1 任务图节点拖不动:护栏 `test_quest_graph_layout_persist` 直接手调 `node.setPos()+on_moved()`,绕过了 `mousePressEvent` 吞掉左键这一真实断点,6 例全绿而特性对用户完全不可用。
- P1-01/02 出生点删/增复活:护栏测的是 staging model 往返,测不到"有 pending 编辑时删/增走了直写源、被 stale 快照盲提交覆盖"的流程序列。

**根因**：测试在"最方便打桩的层"验证,而不是在"用户实际触发的层"。手调 setPos 测了数据管线、没测鼠标管线;model 往返测了序列化、没测门控时序。

**这次补的护栏**：
- P0-1 配了真实 `QMouseEvent` press→move→release 打到 view 上的回归测试。
- P1-01/02 配了"编辑→切走/删/增→断言模型"流程探针。

**今后硬规则**：
> **交互特性的护栏必须从"用户触发的最外层入口"进,不许从中间函数打桩。** 拖拽测 QMouseEvent 不测 setPos;快捷键测 keyPressEvent 不测直调 handler;门控测"编辑→操作→断言模型",不测单个 commit 函数。一条测试若"手动把系统摆到断言点",它证明不了断点之前的路能走通。

---

### 根因2：手工镜像清单没有语义级 parity 测试

**表现**（这次一口气抓到 5 处漂移,全是同一个病）：
- P1-10 `_PARAM_SCHEMAS`(编辑器)↔`actionParamManifest.ts`(运行时权威):25 处"Python 当必填/TS 可选"倒挂,把合法数据拦到存不了。
- P1-12 `CONTENT_ID_PARAMS`(登记 id 参数)↔控件生成链:3 个登记了 id 却还是裸文本框。
- P1-15 `KNOWN_DIRTY_BUCKETS`↔`lsp_client.overlay_payloads`:characterRegistry 漏镜像,搜索承诺实时却给旧值。
- signal_refactor 两张登记表:注释宣称"有 parity 测试"实际不存在。
- P1-03 重构引擎 speaker 通道↔运行时真实数据形状:`kind=="sceneNpc"` 整条漏。

**根因**：项目有 N 处"必须与另一处保持同步"的手工清单,但 parity 测试只锁"存在性"(两边键集合相等),不锁"语义"(required 子集关系、控件类型、序列化形状)。而且有的清单靠**注释**声称有护栏——注释不是护栏。

**这次补的护栏**：
- required 改从 TS manifest 单一真相源读(`narrative_required_params.py`),配"Python required ⊆ TS required"语义 parity。
- CONTENT_ID_PARAMS 33 项逐一离屏构造断言"不是裸 QLineEdit"。
- overlay 镜像↔KNOWN_DIRTY_BUCKETS 每个单文件内容桶必须在镜像表或显式豁免集合。
- signal_refactor 两表真补上 parity 测试。

**今后硬规则**：
> **每一处手工镜像清单都必须有语义级 parity 测试,且宁可消灭镜像(读单一真相源)也不要维护两份。** 新增 action/数据域/id 参数时,三/四/五件套要连 parity 测试一起提交。**注释里写"有护栏"= 没有护栏**;声称的护栏必须能 grep 到对应测试。

---

### 根因3：引用生命周期(rename/delete)防护不成体系

**表现**：同一工作区里引用防护有三个档次并存——entity_refactor 引擎(全跟随+报告+撤销)、character/smell(计数警告或拒删)、**flag/plane/quest id/rule 碎片/出生点 key 完全裸奔**。且面板 id 输入框裸改可完全绕过重构引擎。恰恰 flag 是历史 P0 的域(P1-13 一键就能重造"未登记 flag 被 bool 化")。

**根因**：改名/删除防护是每个编辑器各自为政加的,没有统一"引用生命周期"契约;谁想起来加谁加。

**这次补的护栏**：flag/plane/quest 改名删除都接了全工程引用扫描(口径对齐 validator 遍历)+确认/跟随改写;validator 补了 cutscene 重复 id、updateQuest.id 悬垂检查。

**今后硬规则**：
> **任何"实体/id/key 的 rename 或 delete"入口,提交前必须回答三个问题:①谁引用它(全工程扫描)?②改名跟随改写还是拦?③删除拦/警告/连带?** 裸 `QLineEdit` 改"引用他者的 id"是禁止的(必须走选择器或重构菜单)。新增可被引用的实体类型时,同步登记到引用扫描面 + validator。

---

### 根因4：fail-open 惯性——"取不到状态就当没事"

**表现**：
- 叙事 Web 无 ErrorBoundary:组件崩→白屏→flush 拿不到数据却返回 True→Save All 假成功。
- 脏检查超时→当"不脏"→自动重载丢草稿。
- `_nav_hit_generic` 不看 select_by_id 结果就报"已定位",聚光可亮在错误条目上。
- 音频试听失败全静默;F5 无脏检查直接 reload。

**根因**：异常/超时/未知返回值的默认分支被写成了"乐观放行",而这些路径的代价是静默数据丢失。

**这次补的护栏**：flush 拿不到数据返回 False+可读错误;脏检查超时当脏(fail-safe);select_by_id 全部返回 bool、导航按真实结果诚实播报;ErrorBoundary 兜崩溃并保留草稿;F5/beforeunload 加脏确认。

**今后硬规则**：
> **凡是"拿不到状态/超时/异常"的分支,默认必须是 fail-safe(当作脏、当作失败、如实报错),绝不 fail-open(当作干净、当作成功、静默放行)。** 判据:如果这个分支猜错了,代价是不是"静默丢用户数据"?是就必须保守。任何"成功"回报都必须基于真实结果,不能基于"函数被调用了"。

---

### 根因5：Save All 连坐面过大——一个角落的错拖垮全局保存

**表现**：
- P1-11 图对话面板 flush 失败直接 raise,中断整个 save_all,其余编辑器改动全没落盘、后续编辑器连 flush 都没执行。
- presave 引用/scenarios 校验不按脏桶收口:盘上既有的无关坏数据锁死一切保存。
- plane extends 成环、water 坏 id 都拖到 Save All 才爆且全域被拦。

**根因**：Save All 被设计成"全有全无"的原子操作(这对写盘阶段是对的),但把"单面板 flush 失败"和"无关域的既有坏数据"也纳入了同一个否决链。

**这次补的护栏**：Save All 单面板 flush 失败降级为"跳过该面板+明示",不中断全局;presave 校验按脏桶收口(只校验将写盘的域);编辑时刻即时反馈(plane 环红字、water id 即时查重),不拖到保存。

**今后硬规则**：
> **区分"写盘原子性"和"校验否决面"。** 写盘要全有全无(StagedJsonWriter 已做对);但校验只能否决"本次将写盘的脏域",不许让无关域的既有坏数据或单个面板的问题拦下全局保存。错误要在编辑时刻就地反馈,不要攒到 Save All 才一次性爆。

---

### 根因6：协作模型(agent 写盘 + 编辑器常开)零防护

**表现**：P1-19 编辑器开着时 agent 改了 JSON,Save All 零检测 last-writer-wins 静默覆盖。这是本项目**标准工作流**(CLAUDE.md 明写"agent 直接改 JSON,人类通过编辑器维护"),不是边缘场景。全库唯一有基线比对的是图对话。

**根因**：编辑器的数据模型假设"我是唯一写者",但项目实际是双写者。

**这次补的护栏**：ProjectModel 载入/保存记 (mtime_ns,size) 基线,`detect_external_changes()` 在 Save All 前检出被外部改动的文件,主窗弹"覆盖/取消"。

**今后硬规则**：
> **编辑器不得假设自己是唯一写者。** 任何全文件粒度的写盘,提交前必须比对基线;发现外部修改必须让用户知情选择,不许静默覆盖。新增数据域接入 save_all 时,一并接入外部修改检测基线。

## 落到流程上的四条收口(给下一个改编辑器的人)

1. **改交互特性 → 护栏从最外层入口进**(QMouseEvent/keyPressEvent/"编辑→操作→断言模型"),不许手调中间函数。
2. **碰任何镜像清单 → 连语义 parity 测试一起交**,能读单一真相源就别维护第二份;注释声称的护栏必须能 grep 到。
3. **加 rename/delete 入口 → 先答"谁引用它/跟随还是拦/删除怎么办"三问**;引用他者 id 一律选择器。
4. **异常/超时/未知返回 → 默认 fail-safe**;Save All 只否决脏域、单面板失败只跳过该面板。

## 验收门升级建议(收尾时并入 agent_docs/editor-tools/norms.md 与 recipes/editor-change-verification-gate.md)

原三件套(全量测试+素材审计+validate-data)对本轮问题**全部无感**。补两条:
- **流程探针门**:凡改门控/切换/撤销/删除/新增路径,必须有"从用户入口触发→断言模型"的探针,而非 model 往返。
- **镜像 parity 门**:CI 扫描项目内所有"必须与另一处同步"的清单,缺 parity 测试即失败(把 test_action_manifest_parity / test_dirty_bucket_parity 的模式推广到全部镜像面)。
