# Godot 运行壳完全对齐差异审计

> 日期：2026-07-12  
> 权威运行时：TypeScript / Pixi 版本  
> 被审计运行时：`godot_port/`  
> 审计方式：只读静态对照 + 现有测试/验收门检查 + 当前双壳快照实跑；未修改运行时代码。  
> 判定：**未达到“同一份资源与 JSON、同一输入序列、同一最终结果”的可用门槛，不得标记完全对齐。**

## 1. 结论先行

当前 Godot 壳不只是“还有少量表现没抛光”，而是同时存在四类阻断问题：

1. **现有生产 JSON 已触发确定性分叉**：例如 `码头白天.json` 的 `fadingZoom{zoom:1.5}` 在 TypeScript 渐变到 `1.5`，Godot 因读取错误参数键而渐变到 `1.0`。
2. **同一个存档会恢复出不同状态**：Godot 读档会补跑旧时间线 zone 的 `onExit`，也会在 scenario/narrative 尚未恢复时提前重评任务和档案。
3. **真实玩家路径不完整**：菜单、地图、背包、规矩使用、书架/档案等大部分面板只有文字外壳，测试通过 `debug_*` 直调绕过了实际不可操作的问题。
4. **现有“零差异/101 项完成”验收不能证明语义等价**：差分工具只比较 `dev_room` 启动后的一帧快照，没有向两端执行同一操作序列，也没有观测事件顺序、存读档副作用、随机流、UI 输入和逐帧状态。

本轮已经找到多项可复现的 P0/P1 差异；任意一项都足以否决“完全对齐”。因此，当前导出包可以作为迁移开发预览，但不能作为与原运行时可互换的资源/JSON 运行壳。

## 2. “field differences: 0” 为什么不能作为结论

2026-07-12 重新执行：

```bash
python3 godot_port/tools/parity_runner.py run \
  --require-equal \
  --report /tmp/gamedraft-current-parity.json
```

结果仍为：两端 schema OK、`field differences: 0`。但这个结果只代表一个极窄检查点相同。

### 2.1 没有共同操作序列

- Godot 请求固定只有 `ping` 和 `captureSnapshot`：`godot_port/tools/parity_runner.py:211-258`。
- TypeScript 侧也只投递一次 `captureSnapshot`：`godot_port/tools/parity_runner.py:271-320`。
- Godot 桥只实现上述两个 operation：`godot_port/scripts/core/runtime_command_bridge.gd:43-59`。
- 权威命令契约实际有 31 个命令，但 `setFlag`、切场、对话推进、选择、玩家输入、存读档等都不能被同一套 parity 请求在两端执行。

所以当前工具证明的是：**两个壳都能启动到特制的 `dev_room`，并能各自拍一张形状相似的照片**；它没有证明任何状态迁移。

### 2.2 快照本身有大面积盲区

共享快照只包含 flags、quest、scenario、narrative、当前对话、玩家、位面、背包和少量 playerView：

- TypeScript 快照入口：`src/core/Game.ts:3451-3477`。
- Godot 快照入口：`godot_port/scripts/bootstrap.gd:343-361`。
- Schema 的大部分对象仅声明 `type: object` 且 `additionalProperties: true`，内部字段没有强约束：`godot_port/compatibility/runtime-snapshot-schema.json:53-83`。

未进入共享快照的关键状态包括：

- Day 的延迟事件队列；
- Health、Smell 分层状态；
- Rules、Archive 的 read/firstView 状态；
- sceneMemory 全量内容；
- active zone、zone action 队列；
- 音频集合与 fade 中间值；
- cutscene/encounter/minigame 在途状态；
- UI 返回栈、面板按钮可达性；
- 对话日志、真实游玩时间；
- 事件序列和 action 失败/继续语义；
- RNG seed/消费位置。

即使以后补一串命令，若快照不扩展，上述分叉仍可完全不可见。

### 2.3 “101/101 完成”是声明集合相等，不是自动语义验证

`audit_runtime_coverage.py` 从 TypeScript 提取能力名，再读取人工维护的 `capabilities.json`：`godot_port/tools/audit_runtime_coverage.py:503-519`。最终所谓 verified 只是 `required & claimed` 的集合计数：`godot_port/tools/audit_runtime_coverage.py:522-590`。

它没有验证：

- Godot handler 是否读取相同参数；
- 默认值、归一化、拒绝和异常语义是否相同；
- 对应测试是否存在、是否走真实路径；
- 测试是否与 TypeScript 使用同一输入和同一 oracle。

因此 `fadingZoom` 虽被计作“已验证”，当前生产数据仍已跑偏。

### 2.4 全套测试的 full parity 默认也不要求相等

`run_tests.py --full-parity` 调用 `parity_runner.py run` 时没有传 `--require-equal`，并只检查输出中出现 `field differences:`：`godot_port/tools/run_tests.py:161-165`。因此即使报告存在非零差异，整套测试仍可能通过。

### 2.5 当前“存档互通”只证明 JSON 信封能搬运

- 测试中的 TypeScript collector/distributor 只是对任意对象做 `structuredClone`：`godot_port/tools/save_interop_probe.test.ts:18-40`，并没有启动真实 `Game` 和真实系统反序列化。
- E2E 只检查一个 flag 被修改，以及系统桶名称没有丢失：`godot_port/tools/save_interop_e2e.py:41-47`。
- 它没有逐桶比较恢复后的真实系统状态、事件副作用、sceneMemory、对话日志、游玩时间和再次保存后的规范化载荷。

此外，产品层面的“导出/导入”也尚未闭环：Godot `SaveManager` 有底层 `export_slot_payload/import_slot_payload`（`godot_port/scripts/runtime/save_manager.gd:124-133`），但菜单没有入口；TypeScript `SaveManager` 没有对应产品 API。当前互通能力只存在于测试脚本，不是玩家可用功能。

## 3. 已确认的状态/玩法分叉

### GD-P0-01：现有 `fadingZoom` 参数键读错

证据：

- TypeScript 权威读取 `p.zoom`：`src/core/ActionRegistry.ts:964-971`。
- Godot 读取 `params.scale`，缺省 `1.0`：`godot_port/scripts/runtime/action_executor.gd:364`。
- 当前生产数据写的是 `zoom: 1.5`：`public/assets/scenes/码头白天.json:698-702`。

实际结果：同一段码头演出，TypeScript 相机渐变到 `1.5`，Godot 渐变到 `1.0`。这是当前内容已经触发的确定性差异。

### GD-P0-02：读档会把旧 zone 的 `onExit` 写进新时间线

证据：

- TypeScript 的存档专用重载会先静默清理 active zone：`src/core/Game.ts:752-761`、`src/systems/ZoneSystem.ts:100-113`。
- Godot 已有 `clear_active_zones_for_restore()`，但读档重载没有调用：`godot_port/scripts/systems/zone_system.gd:91`、`godot_port/scripts/bootstrap.gd:599-600`。
- Godot 卸场景会把 zones 设为空并正常触发 exit：`godot_port/scripts/systems/scene_manager.gd:113-124`。

最小复现：当前局位于 zone Z，Z 的 `onExit` 写 `x=true`；目标档中 `x=false`。TypeScript 读完保持 `false`；Godot 先分发存档，再卸旧场景，旧 Z 的退出动作会把新时间线改成 `true`。

### GD-P0-03：Quest/Archive 在半恢复状态下提前重评

证据：

- TypeScript 在整个恢复事务外层启用/关闭 restoring guard：`src/core/Game.ts:2939-2959`。
- Godot 的分发没有调用两个 manager 已实现的 `set_restoring`：`godot_port/scripts/bootstrap.gd:591-596`。
- Godot 恢复顺序中 quest 早于 scenario/narrative：`godot_port/scripts/bootstrap.gd:119-133`。
- Quest deserialize 同步 flag，listener 会立刻 evaluate：`godot_port/scripts/systems/quest_manager.gd:140-151,175-177`。

最小复现：旧局叙事 G 已到 Done；目标档为 G=Start、任务 Q=Inactive，Q 的完成条件依赖 reached Done。Godot 恢复 Q 时仍看到旧 G=Done，会先把 Q 完成，之后才把 G 恢复到 Start；TypeScript 不会在半态求值。

### GD-P0-04：reactive 迁移与 action 批的调度顺序不同

- TypeScript `flag:changed` 通过 `queueMicrotask` 合批：`src/core/NarrativeStateManager.ts:358-371`。
- TypeScript 每条 action 都经过 Promise await 边界：`src/core/ActionExecutor.ts:176-195`。
- Godot 使用 `call_deferred` 到后续 idle/frame：`godot_port/scripts/systems/narrative_state_manager.gd:364-376`。

最小复现：G 初态 A；`x=true` 的 reactive 为 A→B；信号 `go` 为 A→C；执行 `[setFlag x=true, emitNarrativeSignal go]`。TypeScript 先 reactive 到 B，Godot 先消费 `go` 到 C，最终状态不同。

### GD-P0-05：水上小游戏每日降级过滤完全相反

- TypeScript 在 degraded 状态剔除 premium、保留普通实体：`src/systems/waterMinigame/WaterMinigameScene.ts:269-272`。
- Godot 在 degraded 状态只保留 premium：`godot_port/scripts/minigames/water_scene.gd:91-94`。
- 每日软上限状态入口：`src/systems/waterMinigame/WaterMinigameManager.ts:101-117`、`godot_port/scripts/minigames/water_manager.gd:47-61`。

实际结果：同一个存档达到第 4 次后，两端生成的是相反实体组，玩法结果必然不同。

### GD-P0-06：真实玩家 UI 大片不可操作

Godot 的通用面板只创建标题和 `RichTextLabel`：`godot_port/scripts/ui/runtime_text_panel.gd:36-43`。下列关键操作只暴露给调试方法，没有按钮/键盘选择路径：

| 面板 | Godot 证据 | 缺失的真实路径 |
|---|---|---|
| 菜单 | `godot_port/scripts/ui/menu_ui.gd:11-26` | 新游戏、继续、存档、读档、设置、返回主菜单 |
| 地图 | `godot_port/scripts/ui/map_ui.gd:20-36` | 点击地点旅行 |
| 背包 | `godot_port/scripts/ui/inventory_ui.gd:11-23` | 选中物品、查看详情、丢弃 |
| 规矩使用 | `godot_port/scripts/ui/rule_use_ui.gd:15-21` | 选择可用规矩 |
| 书架/档案 | `bookshelf_ui.gd:15-31`、`archive_entry_panel.gd:26-49`、`book_reader_ui.gd:14-32` | 打开分类、选择条目、翻页/跳条目 |

TypeScript 对应实现有实际 pointer 控件：`src/ui/MenuUI.ts:102-350`、`src/ui/MapUI.ts:387-467`、`src/ui/InventoryUI.ts:151-225`、`src/ui/RuleUseUI.ts:132-188`。

现有 Godot 测试直接调用 `debug_travel/debug_mode/debug_select/debug_discard_selected/debug_open/debug_navigate`：`godot_port/tests/hud_map_menu_test.gd:12-17`、`runtime_panels_test.gd:15-17`、`archive_ui_test.gd:7-13`。所以测试通过恰恰绕过了真实不可玩路径。

### GD-P0-07：地图旅行语义绕过位面门闸和 overlay 栈

- Godot 注册地图时没有 openGuard：`godot_port/scripts/bootstrap.gd:284-290`。
- Godot 收到 travel 后直接 `set_state(Exploring)` 并调用 SceneManager：`godot_port/scripts/bootstrap.gd:666-669`。
- TypeScript 在打开前和旅行时有两道位面门闸，并经 ActionExecutor 切场、正确恢复 overlay 栈：`src/core/Game.ts:2402-2419`、`src/core/EventBridge.ts:99-119`。

位面禁用旅行或地图打开后位面改变时，Godot 仍可能旅行；直接 set state 还会留下旧 overlay 返回栈，后续面板关闭可恢复到错误状态。

### GD-P0-08：规矩结算漏写 `rule_used_*`

- Godot 只执行 resultActions，并把 resultText 变为 toast：`godot_port/scripts/bootstrap.gd:650-654`。
- TypeScript 还会写 `rule_used_<ruleId>`，并以阻塞 InspectBox 展示结果：`src/core/EventBridge.ts:152-169`。

同一规矩 JSON 在 Godot 不留下权威进度 flag，后续条件和事件可分叉。

## 5. 存档、互通和导出仍未对齐

### 5.1 载荷字段缺失

- TypeScript 保存 `dialogueLog` 和真实 `playTimeMs`：`src/core/Game.ts:2927-2956`。
- Godot 不保存/恢复对话日志，且把 `playTimeMs` 固定为 0：`godot_port/scripts/bootstrap.gd:584-596`。
- Godot 的 `dialogue_log_ui` 明明已有 `serialize/deserialize`，但未接线：`godot_port/scripts/ui/dialogue_log_ui.gd:16-17`。

后果：

- TS→Godot：日志字段被忽略，再保存时消失；游玩时间变 0。
- Godot→TS：载荷没有 dialogueLog，TypeScript distributor 因 key 不存在而可能保留当前会话旧日志，不是目标档状态。
- 两边菜单元数据的游玩时长不可能一致。

### 5.2 产品导出/导入不可达

- Godot 只有底层函数，无菜单按钮或文件选择流程。
- TypeScript SaveManager 没有导出/导入产品 API。
- 当前所谓双向流程依赖测试进程读取本地临时文件，不是游戏内能力。

所以用户已经确认的“能导出、要互通”目前只完成了测试线上的 JSON 搬运，不是可交付功能。

## 7. 画面、过场和音频不是同一实现

这些并非主观“看起来略有不同”，而是公式/采样/时序不同。

### 7.1 当前启用的实体照明/深度管线

## 8. 其它确认差异与当前未必触发的边界

- 当前档案/书籍包含 `[img:...]`，TypeScript 用 `RichContent` 加载图片（`src/ui/RichContent.ts:30-101`）；Godot 通用面板关闭 BBCode 且无图片解析（`runtime_text_panel.gd:36-43`），会显示成文字。
- Godot InspectBox 固定 120 高度（`inspect_box.gd:21-29`），TypeScript 按正文动态扩展（`src/ui/InspectBox.ts:35-68`），长文会被截断。
- TypeScript ActionChoice 支持 `Digit*` 和 `Numpad*`（`ActionChoiceUI.ts:123-138`）；Godot 只匹配 Digit，InputManager 也没有小键盘数字映射（`action_choice_ui.gd:40-42`、`input_manager.gd:240-259`）。
- 图对话 deferred chain：TypeScript 有 chain runner 防重入和最终结束锁（`GraphDialogueManager.ts:649-712`）；Godot 没有同等结构（`graph_dialogue_manager.gd:101-104,295-298`）。当前尚未证明有效 JSON 形成多层同步结束链。
- 禁用且带金币成本的选项，其 disable hint 优先级不同：`GraphDialogueManager.ts:894-920` vs `graph_dialogue_manager.gd:183-194`。
- 已有 scripted dialogue/encounter 时再次 start：TypeScript 可覆盖，Godot 拒绝；见 `DialogueManager.ts:48-63` vs `dialogue_manager.gd:17-18`、`EncounterManager.ts:96-109` vs `encounter_manager.gd:42-45`。
- Minigame session/action gate 与 Cutscene 整体缺少 TypeScript 等价的 catch/finally teardown；损坏资源或 action 异常时可能残留 active、输入锁、staging 或 playing。见 `minigameSession.ts:57-67,215-297` vs `minigame_session.gd:67-115`、`minigame_action_playback_gate.gd:20-32`，以及 `CutsceneManager.ts:448-514` vs `cutscene_manager.gd:89-107`。

## 11. 最终验收门槛

只有同时满足以下条件，才可以写“完全对齐”：

- **逻辑状态：零容忍**。同一 canonical 初态 + 同一命令/输入序列后，所有系统状态、事件顺序、action 结果、存档载荷完全一致。
- **失败语义：零容忍**。两端必须在同一 action 停止/继续，并返回同一错误类别；禁止一端静默成功。
- **存档：双向真实互通**。两端真实 Game 至少完成 TS→Godot→TS、Godot→TS→Godot，多检查点继续运行后仍一致；导出/导入是玩家可达功能。
- **时间和随机：可复现**。固定 seed、fixed tick、同一 RNG 消费序列；失焦和暂停后也能恢复一致。
- **玩家路径：只准真实输入**。菜单、地图、背包、对话、规矩、档案和小游戏验收不得用 `debug_*` 代替按钮/键盘/触屏。
- **表现：golden 验收**。每个场景/过场/小游戏有截图或帧序列对照；视觉允许的误差必须事先量化，不能用“差不多”替代。若要求逐像素完全相同，则必须采用共享渲染算法并锁定平台/GPU；跨引擎、跨 GPU 的位级像素相等不能靠当前实现保证。
- **门禁不可绕过**。coverage 必须由 handler/契约/测试映射自动生成；full parity 任意差异直接失败；禁止人工把 capability 名写进 JSON 就计为 verified。
