# Godot 代码级直译总账（2026-07-13）

## 结论

这不是“按测试补功能”的移植，也不是“Godot 重新设计一套近似架构”。唯一可收敛的方法是：以 TypeScript 运行时源码为逐文件模板，把同一模块原样翻译成 GDScript；Godot 只负责语言语法、渲染、输入、音频、文件加载和事件循环等平台机械替换。

机器可读的逐文件总账在 `godot_port/compatibility/code-translation-contract.json`。结构审计会拒绝漏登记的可执行 TypeScript 模块、失效目标路径以及已禁止的 Godot 独有职责。

## 不可变迁移规则

每个源模块必须逐项保留：

1. 文件/模块边界与导出形态；源文件是函数模块，目标不能擅自改成有状态类。
2. 类职责、构造参数、依赖方向和所有权；不能用 service locator、后置钩子或兼容层把职责挪家。
3. 字段集合、默认值、生命周期和字段更新时机。
4. public/private 方法集合、方法顺序、参数和返回/失败语义。
5. 同步/异步边界、调用顺序、事件顺序、批处理短路规则和微任务顺序。
6. JSON 宽松转换语义，包括 `undefined`、`null`、`Number`、`Boolean`、`??`、trim、默认值和非法值处理。
7. 注册顺序、参数清单、存档分区、反序列化顺序和销毁顺序。

禁止事项：

- 禁止为了让单个测试通过，在生产模块中增加源码没有的分组注册、staging、post-action hook 或特殊旁路。
- 禁止让测试桩定义生产架构；测试只能通过唯一生产入口装配。
- 禁止在目标管理器缺方法时绕过管理器直接调用更底层对象；应先翻译源管理器的方法。
- 禁止把“看起来结果一样”当完成；结构、时序和失败路径都必须有机器校验。

## 唯一允许的适配层

适配层只能替换源语言/引擎本身提供、Godot 没有同构 API 的能力，而且不能包含游戏领域判断。

- `queueMicrotask` → `RuntimeMicrotaskQueue`：Godot `call_deferred` 会推迟到后续帧，不能保持 JavaScript 在下一条 awaited action 之前排空微任务的顺序。
- 浏览器 URL/fetch → `RuntimeResourceLocator`/`RuntimeAssetManager`：只替换加载通道，资源相对结构和 JSON 不变。
- Pixi 渲染、浏览器输入、WebAudio → Godot 渲染/输入/音频对象：上层模块仍保留相同调用边界。

每增加一个适配层，必须在机器总账中登记源平台原语、目标文件和不得承载的领域职责。

## 当前校准样板

### ActionExecutor.ts → action_executor.gd

已按源类重排：字段、构造、4 个内建 action、注册/查询、策略栈、单条/批量执行、销毁、`runWithExploreActionLock`。已移除源码没有的 post-action hook 和 lifecycle epoch。

Godot 没有 Promise rejection/exception 通道，当前用 `execute_await() -> bool` 的 `false` 表示被拒绝的 Promise；普通 handler 的业务返回值不得被解释成失败。这个差异必须在后续双运行时失败夹具中继续锁定。

### ActionRegistry.ts → action_registry.gd

已从“有状态类 + 领域注册组 + staging 后重排”改回源文件形态：

- 无实例字段；
- 单一静态 `register_action_handlers(executor, deps)`；
- 97 个 `executor.register` 按 TypeScript 源文件顺序直接出现；
- action 类型和 `paramNames` 逐项结构比较；
- `playScriptedDialogue` 的归一化仍归 ActionRegistry；
- overlay 仍经 CutsceneManager，不越级调用 renderer；
- 测试通过 `tests/support/action_registry_fixture.gd` 调用同一个完整生产入口。

### actionParamManifest.ts → action_param_manifest.gd

已建立同名目标模块，并翻译两个查询函数。DEV 注册审计恢复到 Game/bootstrap 装配层；运行时登记与 manifest 双向比对。

## 迁移顺序

1. `Game.ts` 与构造/初始化/装配链：先固定对象创建顺序、依赖注入、事件桥、系统注册、初始场景提交、tick、保存、销毁。
2. `core` 执行基础：EventBus、FlagStore、GameStateController、InputManager、SaveManager、AssetManager、文本解析、条件解算。
3. 强时序系统：NarrativeStateManager、ScenarioStateManager、SceneManager、InteractionCoordinator、CutsceneManager、GraphDialogueManager。
4. 资源/存档系统：Inventory、Rules、Quest、Archive、Day、DocumentReveal、Plane、Zone、Health、Smell、Audio、SignalCue。
5. 小游戏模块：公共 script/session，再按 pressure-hold、water、sugar-wheel、paper-craft 逐文件翻译。
6. 逻辑全部通过后才对齐渲染、UI、shader、像素密度、动画和音频表现。

不能按“今天修场景、明天修背包”的横向打补丁顺序推进；每次只认一份源文件，读完字段和全部方法后一次性翻译目标模块。

## 单模块完成门槛

模块只有同时满足以下条件，机器总账状态才能从 `audit-required`/`translating` 改成 `verified-direct`：

1. 源/目标文件和导出映射明确，无目标缺失。
2. 字段、构造依赖、方法、注册表及其顺序通过结构审计。
3. 正常、边界、非法参数、失败/取消、嵌套异步、destroy 后回调都有共享夹具。
4. 同一夹具分别驱动 TypeScript 和 Godot，比较最终状态、事件序列和关键中间 trace。
5. 与该模块相连的存档 round-trip 在两端都通过。
6. 不存在未登记的平台差异或目标独有领域职责。

## 当前未对齐风险

机器总账仍明确标记以下事实，不能宣称完成：

- `Game/bootstrap`、Narrative、Archive、SceneManager、CutsceneManager 仍处于翻译中；其中 CutsceneManager 的目标方法规模和源文件差距很大。
- DebugTools、FlagKeys、depthLog、devErrorOverlay、narrativeGraphValidation、holdProgress 还没有一对一目标模块。
- 多数系统已有同名 Godot 文件，但尚未做字段/方法/时序的逐文件审计，只能算“存在候选实现”，不能算直译完成。
- 当前结构审计覆盖核心装配、Action 链、manifest、微任务调度、部分场景/深度/保存顺序；尚未覆盖所有模块方法集合。
- 当前差分 runner 主要比较最终快照，下一阶段要增加逐模块共享输入和事件 trace，避免“最终值碰巧相同”掩盖中间时序差异。

## 收敛判据

最终完成不是“Godot 能跑”，而是：机器总账没有 `missing`、`translating`、`audit-required`；所有非 `verified-direct` 条目均为有依据的纯引擎适配；结构审计、TypeScript 测试、Godot 测试、双运行时差分、存档互通、资源审计和各平台导出全部通过。
