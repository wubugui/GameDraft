# Godot 运行壳架构对齐审计（2026-07-13）

## 验收定义

本轮不再把“同一组快照字段相等”当成逻辑完成。严格对齐同时要求：

1. 同一份资源与 JSON 由两个壳直接消费，不复制第二份业务数据。
2. Action、Condition、事件、状态机、系统生命周期与存档恢复经过同一职责边界。
3. 系统初始化、反序列化、更新与销毁顺序一致。
4. 同输入、同 RNG 状态、同固定步长得到零字段差异；结构门禁失败时，即使当前快照相等也不得宣称完成。

机器可执行契约位于 `godot_port/compatibility/architecture-contract.json`，阻断检查为 `python3 godot_port/tools/audit_architecture_parity.py`。

## 已确认并已修复

| 域 | 原偏差 | 当前结构 |
| --- | --- | --- |
| 系统装配 | Godot 由 RuntimeRoot 工厂构造，且注册顺序与 TS 不同 | bootstrap 显式构造实例；26 个系统按 `Game.registeredSystems` 同序注册；RuntimeRoot 只做生命周期适配 |
| EventBridge | Godot 仅有对话/遭遇桥，商店、背包、地图、菜单、规则使用散落在 bootstrap | `RuntimeEventBridge` 独占与 TS 相同的 17 个事件，并统一注销 |
| InteractionCoordinator | Godot 直接持有 Player/Camera，NPC 对话瞬时缩放 | 依赖形状与 TS 对齐；玩家位置/镜头均走回调；550ms 缩放统一经 CutsceneManager |
| ConditionEvalContext | Godot 在上下文塞入私有 `evaluateList` 回调，且 `currentOwner` 固定为 null | 上下文只保留 TS 字段；消费方统一经 `RuntimeConditionEvalBridge`；owner 使用当前叙事 owner |
| Action 装配入口 | Godot 在 bootstrap 十余处逐段注册 | 生产装配收敛为一次 `RuntimeActionRegistry.register_action_handlers` |

## 尚未通过的严格项

| 优先级 | 偏差 | 风险 | 修复判据 |
| --- | --- | --- | --- |
| P0 | Domain Action handler 的函数体仍物理位于 `RuntimeActionExecutor` | Executor 继续知道 Inventory/Quest/Scene/Cutscene 等具体系统，违背 TS 的 ActionRegistry 边界 | handler 定义迁入 RuntimeActionRegistry；Executor 只保留 4 个 builtin、register/execute/policy/lifecycle |
| P0 | NPC 巡逻状态与推进目前由 Godot SceneManager 持有；TS 由 Game 组合层持有 | 相同动画/场景时序可能因系统更新归属不同而漂移 | 将巡逻会话所有权与动作回调提升到 bootstrap 对等组合层，SceneManager 恢复数据/实体职责 |
| P0 | 需要把状态分支的逐帧调用序列固化为结构契约 | 新系统插入或调用挪位可能不改变短用例快照，却改变长时间结果 | 对 Exploring/Cutscene/Dialogue/Encounter/Minigame/UIOverlay/ActionSequence 与尾部更新顺序建立门禁 |
| P1 | 需要逐个核对异步任务的 epoch/cancel/destroy 对称性 | reload、切场、退出时旧协程可能继续写状态 | 每个持有 await/timer/process_frame 的核心模块均有销毁或代次中止测试 |
| P1 | 需要把架构门禁接入 `run_tests.py` 与完整 parity 套件 | 手动检查容易被后续改动绕过 | 全套入口先跑结构门禁，失败即停止发布/导出 |
