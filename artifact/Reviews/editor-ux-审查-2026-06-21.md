## 各编辑器布局与编辑工作流(审查所建立的理解)

**scene_editor.py**: SceneEditor is a 3-pane splitter: left = QToolBar (+Add Entity menu button, Apply, "● 未应用" dirty label, Delete) over three view-mode checkboxes + a cutscene-context FilterableTypeCombo + an unfiltered QListWidget of scenes; center = SceneCanvas (QGraphicsView) where entities are dragged; right = ScenePropertyPanel, a QScrollArea wrapping a QStackedWidget with one page per entity kind (empty/scene/hotspot/npc/zone/spawn), each page a stack of CollapsibleSection groups built with compact_form. Authoring flow: pick a scene in the list, add entities via toolbar menu or canvas right-click, select an entity (canvas click or drag), edit fields in the right panel, then click Apply to commit staging into the model (auto-discard if you switch entity without Apply). Coordinates/polygons round-trip between side-panel tables and live canvas drag. Most reference fields already use proper selectors (IdRefSelector, FilterableTypeCombo, map/spawn picker dialogs), and the rubric's named "known offenders" (dialogueGraphEntry, inspect entry, patrol move-anim-state) have in fact been converted to model-populated combos.

**narrative_state_editor.py + narrative_data_editors.py**: narrative_state_editor.py is NOT a native PyQt editor: it is a thin QWidget shell (NarrativeStateEditor) that embeds the React-Flow narrative-composition web app in a single QWebEngineView via QWebChannel (NarrativeEditorBridge). All signal/transition/state authoring happens in that web canvas, which is out of scope for a PyQt-widget audit. The only native PyQt UI in that file is the editActions() modal (which correctly reuses the shared ActionEditor) and a placeholder HTML message; there is NO visible PyQt save/reload/dirty toolbar (the _toolbar_save/_toolbar_refresh/_toolbar_reload_page methods exist but are never wired to any button). narrative_data_editors.py contains the two real PyQt editors. ScenariosCatalogEditor and DocumentRevealsEditor both follow the standard left list + right scrollable detail form + bottom 从内存重载/Apply pattern. The author picks a scenario/reveal in the left QListWidget, edits fields in the right form (id via IdRefSelector, phases in a QTableWidget with per-row name QLineEdit + status combo + a requires compound widget, exposes in a flag-key/value table, reveal conditions in a 5-mode QStackedWidget), then clicks Apply to write to ProjectModel. ScenarioRequiresExprEdit is a reusable 3-mode (与/或/JSON) compound used both for the scenario entry-requires and per-phase requires.

**water_minigame_editor.py + water_minigame_canvas.py + paper_craft_editor.py**: water_minigame_editor uses a 3-pane horizontal splitter: a left QListWidget of water-instance ids (新增/删除/预览), a center WaterMinigameSceneCanvas (QGraphicsView where the author drag-places/moves entity sprites and a "点击放置实体" toggle adds new ones), and a right vertical splitter with an instance form (label/spot/surface/bounds/waterBottom) on top and an entity property form below (+实体/−实体, with category/sprite/pos/depth, checkable motion/pull groups, valueTier, cue/hint with custom-vs-Strings-tag stacks, and three ActionEditors). There is NO entity LIST — entities exist only as canvas sprites, so authoring is canvas-driven. paper_craft_editor is one flat panel: a top instance QComboBox + 预览, then a 2-column body where every sub-collection (订单/部件/槽位/纸色/收尾) is a QGroupBox whose first row is a QComboBox selector plus a tiny +/− CRUD row, followed by a detail QFormLayout; selecting/adding/removing a sub-item is done entirely through combo boxes. Editing is live (editingFinished/valueChanged write straight to the dict and mark_dirty), so there is no per-field Apply button.

**quest_editor.py + quest_graph_scene.py + quest_graph_items.py + rule_editor.py**: QuestEditor is a 3-pane splitter: left = draggable group/quest tree (drag-reparents); center = a zoomable QGraphicsView showing groups (top level) or quest nodes (after double-click drill-down) with a breadcrumb label and 返回上层/回到顶层/适应视图 buttons; right = a scroll-area property panel that swaps between a group form and a quest form, each built from collapsible sections + a single 应用 (Apply) button. Selecting in the tree, or clicking a graph node, drives the right panel; edits require pressing Apply (with commit-on-leave/Save-All hooks). NextQuests (graph edges) are authored only in the right panel via a per-edge mini-list (_NextQuestsEditor), not by dragging in the graph. RuleEditor is a two-tab (Rules / Fragments) list+detail editor: each tab has a list on the left and a compact_form detail on the right with its own Apply button; the Rules tab also has a linked-fragments sidebar that cross-jumps to the Fragments tab, and Fragments are filtered/owned by a ruleId dropdown rather than a free editable field. The author clicks a list/tree/graph item, edits fields, then clicks Apply per panel.

**archive_editor.py + encounter_editor.py**: Both editors use the project's list+detail+Apply pattern. ArchiveEditor is a QTabWidget with 4 tabs (Characters/Lore/Documents/Books); each tab is a left QListWidget (+Add/Delete) plus a right QScrollArea detail form ending in an Apply button. The author picks an item on the left, edits the form on the right, and must click Apply to write back. The Books tab is the deepest: a single scroll column stacks the book form, then a Pages QListWidget with its own add/del/up/down buttons, then a page form, then a Page-entries QListWidget with its own buttons, then the entry form — three nested list levels in one vertical scroll, all governed by one bottom Apply. EncounterEditor is a single list+detail: pick an encounter, edit id (IdRefSelector + "生成唯一 id")/narrative/options, where each option is a collapsible OptionWidget (default-collapsed) holding text/type/rule/result and collapsible Conditions/Consume/Actions sub-sections, then Apply. Encounter guards unsaved edits on row-switch via a save/discard/cancel prompt; Archive does not.

**item_editor.py + shop_editor.py + game_config_editor.py + overlay_images_editor.py + player_avatar_editor.py**: All five are master-detail or single-form editors on the established list+detail+Apply pattern. item_editor/shop_editor use a left QListWidget (+Item/Delete) plus a right detail form with a manual Apply button; both implement commit-on-leave + confirm_close so edits aren't silently lost. game_config_editor and player_avatar_editor are single-form panels (no list) with one Apply button; overlay_images_editor is a flat scrollable list of '短id + image-path + delete' rows with one Apply. The author workflow: pick/add an entity, type into fields / pick from selectors / edit sub-lists (dynamicDescriptions, shop items table, startupFlags table), press Apply to push into the in-memory ProjectModel, then Ctrl+S to write JSON. Top-level selectors are used well (shop itemId via IdRefSelector, config scene/quest/cutscene refs, flags via FlagKeyPickField/FlagValueEdit, avatar bundle combo, overlay paths via CutsceneImagePathRow; item type is the closed enum consumable|key so a QComboBox is correct), so the remaining gaps are sub-list editing affordances, a number-as-free-text cell, and grouping/reorder/discoverability rough edges rather than missing top-level pickers.

**string_editor.py + audio_editor.py + anim_editor.py + filter_editor.py + flag_registry_editor.py + action_registry_editor.py**: These six editors split into two interaction styles. (1) List/detail browsers — string_editor (a 2-column QTreeWidget of categories→keys with a detail form below: key field, value-type combo, and a stacked RichText/number/bool value widget), filter_editor and anim_editor (left QListWidget + right detail form), and action_registry_editor (filter row + QTableWidget + read-only detail labels). The author selects an item on the left/top and edits fields on the right/bottom, then clicks a per-panel 'Apply' to push into the model. (2) Table/row editors — audio_editor (a QTabWidget of BGM/Ambient/SFX/System-SFX, each a QTableWidget with +/− Entry and Apply) and flag_registry_editor (a static-flags QListWidget on the left plus a scrollable stack of per-pattern QFrame rows on the right). To author content the user types ids/values into cells or form fields, picks types from combos, uses the few selectors present (audio sfx-id IdRefSelector, RichText value field), and must remember to click Apply per tab/panel; heavy preview/external-tool work (anim spritesheets via video_to_atlas, filter matrices via filter_tool) is delegated to separate launched tools. The dominant friction themes are missing selectors on enumerable keys, missing search/context-menu/reorder on lists that can grow large, structured numeric data edited as raw text, and inconsistent Apply/dirty/labeling conventions across panels.

**map_editor.py + game_browser.py + dialogue_graph_editor_tab.py + pressure_signal_editor.py**: All four follow the project's list+detail pattern but with two different commit styles. map_editor (3-pane splitter: node list / draggable canvas / props scroll) and the dialogue graph tab use LIVE commit — every field write-through immediately, no Apply button; map nodes are positioned by dragging circles on a zoomable QGraphicsView (x/y spinboxes mirror the drag). pressure_signal_editor (PressureHoldEditor + SignalCueEditor, list + scrolled detail) uses an explicit Apply button plus commit-on-leave/flush hooks. An author authors content by: picking a list row, editing form fields (id/name/scene/coords/conditions/actions), and for pressure holds adding interrupt sub-blocks (atRatio/resetToRatio/abort + an ActionEditor) and an onComplete ActionEditor. game_browser is a non-data toolbar + embedded WebEngine game preview (Run/Reload/External + read-only URL). dialogue_graph_editor_tab is a thin wrapper embedding the full DialogueGraphEditorWidget (toolbar, dual filtered lists, OdenGraphQt flow canvas, collapsible graph-properties form, node inspector).

**shared/ widgets**: These are reusable building blocks wired into nearly every editor. The dominant pattern is a vertical list of rows + "+ Add" button at the bottom (ActionEditor, ConditionEditor, ScriptedLinesEditor, RuleSlotsParamEditor, ActionChoiceOptionsEditor). An author picks an Action type via ActionTypePickerField (a read-only line + "选择…" button opening a searchable modal list with a red-dot legend for save-mutating actions), then edits a QFormLayout of params that rebuilds when the type changes. Conditions are authored two ways in ConditionEditor: a flat list of flag rows (AND group) plus a recursive ConditionExprTree (all/any/not + flag/quest/scenario/scenarioLine/narrative leaves) inside a scroll area, with a default-collapsed expert raw-JSON fallback. Referenced values mostly go through selectors (IdRefSelector, FlagKeyPickField→FlagPickerDialog modal, FilterableTypeCombo, HexColorPickRow, CutsceneImagePathRow), and player-visible text goes through RichTextLineEdit/RichTextTextEdit with an "引用" button that opens InsertRefDialog. The friction concentrates in: a few leaf fields that still use bare QLineEdit instead of a selector, collapsed action rows showing no summary of their contents, modal-heavy flag/ref picking, and inconsistent reorder/affordance treatment across the otherwise-similar list editors.

**cross-editor consistency + main_window navigation/tab structure**: The app is a QMainWindow with a left QTreeWidget nav + right QStackedWidget (one page per editor). _populate_tabs() builds a fixed list of ~28 editors grouped under hardcoded tree paths (物理世界 / 数据编辑 > 叙事编排 / 规则与经济 / 注册表与扩展 / 资源与本地化 / 工程与全局 / 运行与预览). Authoring is two-staged everywhere: inside each editor the author edits a list+detail panel and clicks a per-panel "Apply" button to write into the in-memory ProjectModel (marking it dirty); then a global "Save All" (Ctrl+Shift+S, also a menu-bar corner button) flushes every editor to model and writes JSON to disk. Run/Stop game and Validate live in the menu bar, a menu-bar corner widget, and (run/stop) again on the Game tab. Most editors follow a list+detail+Apply convention, but button labels, Apply labels, dirty indicators, list affordances (search/reorder/context-menu), and the meaning of the Edit>Save action vary widely between editors.
