## 〇、系统理解（审查前提，供后来者快速建立心智模型）

**五层信号脊椎**（权威卡 `agent_docs/runtime/mechanisms/narrative-signal-spine.md`）：
世界层（zone/hotspot/npc 挂门闸+发起）→ 对话/动作层（只演+`emitNarrativeSignal`）→ 拍子状态机层（narrative 的 scenario_* 子图消费信号推进，演出放 `onEnterActions`，末态 `broadcastOnEnter` 派生 `state:<图>:<态>` 信号）→ 主线脊椎层（主图纯里程碑，监听子图末态广播线性推进，**唯一进度真相源**）→ 任务清单层（quests.json 纯镜像，completionConditions 用 `reached:true` 叶子）。

**运行时模型**（`NarrativeStateManager`）：
- 所有图常驻激活，每图一个 `activeState` + 单调增长的 `reachedStates` 历史（两者入档，图数据本身不入档）。
- 一切外部输入（信号 / setState 命令 / reactive 命中）进**单队列串行排空**；排空中执行动作期间入队的项由**按深度互斥的嵌套排空**处理（防"动作 await 自己发的信号"死锁）。
- 一次信号对每张图至多应用一条迁移（priority 降序、声明序决胜；提交前复核 from==active 防过期迁移）。
- 迁移应用顺序：from.onExit → 置 active/reached → `narrative:stateChanged`（Quest 镜像、位面对账器同步消费）→ to.onEnter → （若 broadcastOnEnter）入队派生广播。
- reactive\* 迁移无信号，靠条件满足自动触发；重评时机 = 注册后 / 读档后 / `flag:changed`（微任务合批）/ 每轮外层排空清空队列时。振荡兜底 `MAX_DRAIN_STEPS=128`（fail-loud 清队列 + error issue）。
- 存档改名迁移：顶层 `migrations.graphs/states` 单跳映射，deserialize 先重映射再校验，对不上则告警丢弃回退 initialState。
- 校验三方：web 编辑器与运行时**共用同一份** `narrativeGraphValidation.ts`（结构性同源，本轮验证 `editorModel.ts:21` 直接 import）；Python 桥兜底为其刻意子集。

这套设计的正确性支柱：①信号排队串行化（无并发迁移）；②迁移提交前复核；③reactive 自愈（条件式，读档后可补触发）；④reached 单调历史供门控；⑤编辑器/校验闸门把内容错误挡在构建期。**支柱①③有裂缝，见 R2/R3。**

## 二、运行时发现（我方逐条实证/推理）

### R3【红线·已实证】存档不含叙事队列在飞项，signal 型迁移读档后无法补发 → 主线永久卡死

- **位置**：`serialize()`（NarrativeStateManager.ts:557）只存 `activeStates + reachedStates`；队列/在飞排空不入档。`deserialize` 后 `kickReactiveEvaluation` 只能自愈 **reactive** 迁移；signal 型迁移（现网 137 条中 133 条，主线里程碑 11 条全是）错过即永失。
- **实证**（临时 vitest，已删）：子图 W 进入末态 w1（broadcastOnEnter）→ onEnter 动作挂起期间取 `serialize()` 快照 → 该档恢复后 W=w1、主图 M 永远停在 m0——`state:W:w1` 广播只在"进入那一刻"发，读档不再补发。
- **现实触发面（经排查被两道门控压窄，但未闭合）**：`canSave` 只许 Exploring/UIOverlay（Game.ts:795）；且每条动作执行期把 Exploring 锁成 ActionSequence（ActionExecutor.ts:218 `runWithExploreActionLock`），面板开启守卫又要求 Exploring（GameStateController `togglePanel`）→ 演出类动作期间玩家开不了菜单。**残余通道**：①UIOverlay 中触发的 `flag:changed`→reactive 级联（商店/面板内改 flag），动作在 UIOverlay 态执行不加锁，此刻存档按钮可用；②未来任何自动存档/快捷存档功能；③崩溃/强退不属存档一致性问题但同窗口。当前内容里 narrative 生命周期动作含 `waitMs`×2、`waitClickContinue`×1（scenario_背尸.try2_face、scenario_河边.took），这些站点本身不带广播故今日实害≈0——**但这是巧合不是保证**：13 个广播态里 6 个带 onEnter 动作（含 3 处 startCutscene），内容侧随时可能把长时动作放进广播态。
- **对照红线**：canSave 注释自陈目标"避免半态存档"（Game.ts:793-794），叙事队列正是漏网的半态。
- **修复方向（建议，未动手）**：最小改 = `canSave` 谓词追加"叙事排空空闲"（`!draining && queue.length===0`，manager 暴露一个 `isIdle()`）；彻底改 = 序列化 pending 队列（外部信号键可重放，广播可由"active 态 broadcastOnEnter 且未被消费"推导，但需消费记账）；防御纵深 = 主线里程碑迁移补 reactive 兜底（`{narrative:子图, state:末态, reached:true}`），模板层可一键带上。

### R2【红线·已实证】deserialize 不清队列、不隔离在飞排空 → 旧时间线写入恢复后的世界

- **位置**：`deserialize()`（NarrativeStateManager.ts:566）只复位/恢复状态映射，不清 `queue`、不使在飞 drain/动作批失效。对照 `registerGraphs` 有 `queue.length=0`、SaveManager 读档专门先 `zoneSystem.clearActiveZonesForRestore()`（Game.ts:783-788）——zone 有旧时间线隔离，叙事没有。
- **实证**：排空挂起在 W.w1 onEnter 期间执行 `deserialize(旧档)` → W 复位 w0、M 复位 m0；旧时间线动作完成后广播补发 → **M 被推进到 m1 而 W 还在 w0**，恢复后的世界被污染成一个两层不一致的状态。
- **对照红线**：norms 不变量 4"旧时间线不写新状态"+ 审批红线"恢复后的状态可能被旧时间线覆盖"，直接违反。
- **触发面**：与 R3 同窗口（读档入口同样受面板守卫限制，UIOverlay 级联窗口 + 未来功能放大）。另一变体：读档后场景重载可能让旧时间线 `moveEntityTo` 的 actor 永不完成 → 在飞排空**永久挂起**（`draining` 恒 true）→ 外层"队列清空后 reactive 重评"永久失效，此后 reactive 迁移只剩 flag 变化能唤醒——一种极难排查的"叙事变迟钝"退化。
- **修复方向**：deserialize 开头做与 destroy 同款的队列 reject + 引入 epoch/generation 计数（在飞 drain 每步比对 generation，不匹配即放弃），或 SaveManager 读档前显式 `narrative.abortInFlight()`。

### R1【中·已实证】嵌套排空使信号到达顺序反转，同信号跨图结果依赖文件序与动作时序

- **机制**：`processTrigger(S)` 按注册序逐图应用；图 A 的 onEnter 动作发出 T 并 await → 嵌套排空把 T **完整**处理（含后续图 C）→ 外层继续把 S 应用到 C。C 视角信号序 = T 先 S 后，与发射序相反。
- **实证**：C 同态挂 S、T 两条迁移，最终 C 停在 `cT`（吃到后发的 T，先发的 S 因 from 不匹配被丢）。若 A 排在 C 之后则结果相反——**同一份数据，语义取决于图在 JSON 里的顺序**。
- **现实影响**：当前内容≈0（脊椎模式下同一信号几乎 1:1 监听、竞争态设计不存在）。开放世界规模下多图共听一信号+监听图动作再发信号的组合会出现，作者没有任何工具能看见这种顺序依赖。
- **建议**：作为**既定语义**写进机制卡（"动作内发射的信号插队完成"），并让编辑器校验对"同信号多图监听且监听图生命周期动作再发信号"的组合给提示；或改运行时为严格 FIFO（把嵌套排空只用于解锁 await、不允许其超车处理后续外部信号——改动大，需权衡）。

### W1【高·代码推理】嵌套排空异常路径不落定已完成项 → 外层排空永久悬挂

- **位置**：`drainAvailableQueue` 的 catch（NarrativeStateManager.ts:885-888）只 `rejectQueuedItems`（队列中未处理项），**不清 `completedQueueItems`**；嵌套循环 `runNestedDrainLoop` 吞异常退出。已处理完毕、等待落定的项（外层动作正 await 它）永不 resolve → 外层 runActions 永不返回 → `draining` 恒 true。
- **触发条件**：嵌套排空中某队列项处理**抛异常**。动作失败被 `runActions` 捕获、EventBus 逐监听器隔离，剩余抛点主要是 `conditionsMet` 里 `conditionCtxFactory()` 本身抛错（未防护，NarrativeStateManager.ts:1001）及未来代码演化。概率低、后果重（违反不变量 3 / 红线"可能永久悬挂的 Promise"）。
- **修复方向**：catch/finally 里补 `resolveCompletedQueueItems()`；顺手给 `conditionCtxFactory()` 调用套 try。

### W2【高·低暴露】`registerGraphs` 清队列不 reject（NarrativeStateManager.ts:336）

`queue.length = 0` 把挂起项静默丢弃，awaiting 方永久悬挂。当前只在启动装载时调用（队列必空），暴露≈0；但与 `destroy()` 的显式 reject 不一致，属同一红线的潜伏点。一行改成复用 `rejectQueuedItems`。

### W3【中·潜伏】reactive 唤醒矩阵有盲点

重评触发源 = flag:changed / 排空尾 / 注册 / 读档 / 注入 ctx。逐叶核对：flag ✓；narrative ✓（状态变化必在排空内）；quest ✓（状态变化必经 `syncFlag` 写 `quest_*_status` 触发 flag:changed，QuestManager.ts:253）；**plane ✗**——`activatePlane` 手动覆盖不发任何可唤醒事件，`{plane:X}` reactive 条件会沉睡到下一个无关信号/flag。当前数据 reactive 条件 100% narrative 叶子（22 处），零现实影响；**新增条件叶子类型时必须同步回答"谁唤醒它"**——建议把这条写进 L2 升级清单的义务里。

### W5【中·校验盲区】`emitNarrativeSignal` 的 `params.signal` 不拦保留前缀

`validateActionDef`（narrativeGraphValidation.ts:900）只查 manifest 必填，不查信号值。内容可写 `emitNarrativeSignal(signal: "state:flow_xungou_main:s12_chufa")` 伪造派生广播直推主线终幕（运行时 `emitNarrativeSignal` 亦不拒 `state:` 前缀，只拒 `__draft__`）。作者信号目录侧有保留字校验（signal.id.reserved），动作参数侧漏。建议校验器（TS + 编辑器 validator.py + 对话图校验）对 emitNarrativeSignal 参数补"保留前缀=error"。

### W4【低】prod 下 narrative_graphs.json 加载失败 = 静默空跑

`loadFromAsset` catch 分支 dev 会 throw，prod 只 warn + 注册空图册（NarrativeStateManager.ts:311-320）→ 所有 narrative 门闸恒 false、信号全部无人听，整个叙事静默死亡（留 recentIssues 痕）。建议 prod 至少给玩家可见的致命错误提示（这不是可降级资产）。

### 其余核查过、无恙/微瑕

- **排空并发**：嵌套排空按深度互斥、外层收尾窗口补启、detached drain 错误路由 recordIssue——设计成熟，B23 类并发循环已防。

- **振荡兜底**：128 步共享计数对当前规模宽裕；触发时 fail-loud 清队列合理。规模化后若单次级联合法超百步需上调（监控项）。

- **重复图 id**：编辑器拒存 + TS 校验 error + 运行时优雅降级三层，✓。
- **id 含 `:`/`|` 拦截**（validateIdDelimiter error）杜绝派生信号键歧义，✓。
- **initialState 的 onEnterActions 不执行**：校验器直接 error 拦（initialState.onEnterActions.unsupported），✓。

- **QuestManager 镜像**：narrative:stateChanged 同步重评 + restoring 抑制半态误判 + 追平逻辑（dev 跳转/读档不一致时非活跃任务直判完成）——注意任务奖励批（questActionTail）与叙事生命周期动作批**并发执行**，奖励只发钱/物时无害，往里放演出会交错（内容纪律项）。
- **死代码**：`canLeaveGraphRemotely` 零引用；`enqueueTriggerKey` 无调用方。清理项。
- **性能**：处理信号 O(全部迁移) 线性扫描、reactive 重评微任务合批、trace/issues 环形上限——当前与 10 倍规模都不是瓶颈。

## 四、编辑器 Python 壳（`narrative_state_editor.py`，43 测试全绿；下列发现均已复现或源码复核）

**健康面先说**：真实盘面字节幂等 ✓（本机独立复验）+ normalize 双跑固定点 ✓；桥协议串与网页正则匹配无破坏；WRAPPER_OWNER 双注册表 10 键对齐；save_all 两阶段写/脏桶登记完整；模板盖章三产物全有全无、零磁盘写；信号注册表不去重（拍板行为）保持。

### P-F1【高·拦合法数据】Python 兜底校验缺 plane 条件叶子 → plane 条件保存必被拦

`_is_condition_shape`（narrative_state_editor.py:2492-2511）没有 plane 分支，`{"plane":"yin"}` 判 False → error `condition.shape` → `save blocked`。TS 权威（narrativeGraphValidation.ts:889-895）明确接受 plane 叶子；编辑器自己桥出去的原生 ConditionEditor 也能作出该叶子（condition_expr_tree.py "激活位面"）。**这正是"Python 兜底绝不能比 TS 严"红线的再犯**（与曾修过的 @owner/@scene 误拦同类，见机制卡）。已实测复现。同类但较轻：quest 缺 status / scenario 缺 status / scenarioLine 缺 lineStatus 时 Python error 而 TS 静默容忍（暴露面主要是 agent 直写）；scenarioSubgraph 缺 graph、wrapper ownerType 空两处也比 TS 严（后者只是幻影 warning）。

### P-F2【高·normalize 代写数据】Python `_normalize_file` 仍在自动补 `broadcastOnEnter: true`，与 2026-07-16 用户裁决相悖

`_apply_derived_broadcast_auto_mark`（narrative_state_editor.py:80-98）对"有 `state:` 监听的状态"强写 `broadcastOnEnter=True`，跑在 getData / saveData / stampTemplate 合并 / save_all 全部路径上。网页侧同款逻辑已按用户裁决（"数据只能由用户改"）**整段删除**（editorModel.ts:118-121 注释自述），改为 TS error `state.broadcast.missing` 引导用户手勾——Python 这份没删，等于把被裁决移除的行为从后门放回：①agent 写入 TS 会报 error 的数据被 Python 静默"修好"落盘；②Python 自己的 `state.broadcast.missing` 检查（:2046-2057）成死代码；③盖章合并路径可能给既有图状态无授权打标。已实测复现（监听未勾广播的状态 → normalize 后被打标、零 issue）。**注**：本轮真实盘面字节幂等能通过，正是因为盘上数据早已被历史 auto-mark 写齐——掩盖而非豁免。

### P-F3【中·休眠雷】信号改名级联/发射目录三张登记表同缺 `archive_lore`

`signal_refactor.py:33-46`（驱动改名级联+使用点预览）与 `narrative_catalog.py:22-35`（驱动 emittedSignals 目录）都没登记档案见闻 `firstViewActions`——它是运行时真实发射面（LoreBookUI.ts:123 执行动作树）。现盘面见闻内 emit 为 0，故休眠；一旦有人在见闻里发信号，改名级联会漏改发射端（监听改了、发射留旧名=运行时静默断线）。parity 测试只锁两表互等，共享盲区自我固化。

## 五、编辑器 Web 应用（React Flow；34 editorModel 测试全绿；下列发现均经代码逐环核实）

**健康面先说**：web 校验直接 import 运行时同一份 `narrativeGraphValidation.ts`（editorModel.ts:20-21），同源非复刻 ✓；两步保存无越过 Save All 的落盘路径；`droppedConditionLeaves` 护栏完好；顶层 migrations 与未知键全程 structuredClone 保留；盖章撞名检测 error 级、失败零副作用；画布模拟器与运行时迁移选择顺序一致；宿主改名自动登记 migrations。

### W-E1【高·跨侧联动】auto-mark 双侧分歧的 web 侧视角（与 P-F2 同根，交叉确认）

web 已按裁决删除 `applyDerivedBroadcastAutoMark`（editorModel.ts:118-121，工作区未提交改动；editorModel.test.ts:797 断言 "normalize never auto-marks"），Python 未删。**放大器**：`applySignalRefactor` 与 `stampTemplate` 两条宿主路径只跑 Python 校验——web Ctrl+S 会被 TS error 拦下的数据，这两条路径被 Python normalize 自动"修好"后放行，还经 `adoptRefactoredNarrative`/`handleStamped` 把被改写的数据回灌画布。用户显式取消的 `broadcastOnEnter:false`（磁盘已有 1 处）只要监听者存在就会被宿主翻回 true。

### W-E2【高·数据一致性】盖章后 Ctrl+Z 击穿"三产物全有全无"，Save All 落盘不一致三件套

`handleStamped`（NarrativeEditorApp.tsx:321-338）把盖章采纳走 `updateData` → 盖章前状态进撤销栈；同类跨文件操作 `adoptRefactoredNarrative`（:344-359）**特意 `resetHistory()`**（注释自述"跨文件重构不允许被画布 Ctrl+Z 半撤销"），盖章路径漏了同款保护。序列：盖章成功 → 画布 Ctrl+Z（叙事回退，quest 镜像/对话桩暂存不动）→ Save All（`flush_to_model` 无条件回读画布覆盖 model.narrative）→ 磁盘出现"有镜像 quest 和对话桩、没有对应叙事编排"的劈叉三件套，narrative 侧校验对 quest 文件无感知拦不住。这正是"全有全无"拍板（agent_docs 决策卡 2026-07-10）要防的产物形态。

### W-E3【中·撤销栈破坏】300ms 合并方向写反：保留了错误端点的快照

`useEditorHistory.ts:17-27` 合并分支 `undoStack[last] = snapshot`（本次编辑前=上次编辑后），正确语义应是合并期间**不动栈顶**（保留连发起点前的快照）。后果：输入一段备注后 Ctrl+Z 只回退最后一个字符，再 Ctrl<-Z 越过整段并**连带撤销上一个独立编辑**；连发起点状态 undo/redo 均不可达。注：已知项"300ms 合并"指机制本身（拍板取舍），本条是实现取错端点，不属已了结范围。另 `wrapUpdater` 先 push 后跑 updater，updater 抛异常留空转撤销步。

### W-E4【中·潜伏】宿主重构引擎（唯一在用的改名路径）漏三个引用面

`signal_refactor.py` 的 rename_state/rename_graph 不级联：① `el.meta.commands`（"图.状态"引用，web 死代码 `updateGraphStateCommandRefs` 证明此面已知需级联，校验仅 warning）；② `el.meta.emits` 里的派生信号声明（signalCatalog 支持却不跟随改名）；③ **`@owner`/`@scene` 相对条件叶**——既不级联、不计数、不告警，且两侧校验器都刻意跳过 `@` 前缀 → 改名后条件运行时永假、全链路无声（运行时支持该 token）。三者当前数据 0 使用，属休眠雷；与 P-F3（archive_lore 漏登记）同类：**重构引擎的引用面登记天生跟不上消费面扩张，缺一张"引用面=消费面"的 parity 清单**。

### W-E5【中】未解析 `{{占位符}}` 可盖进 narrative_graphs（全程仅 warning，TemplatesPanel `canStamp` 只看 ok）——与 P-F9 同项，模块 docstring 自述的"物理隔离"只有 warning 深度。

### 低（记录）

- **W-E6** broadcastOnEnter 取消勾选写 `false` 而非删键（默认键注入，磁盘已有 1 处）；叠加 auto-mark 会被静默翻回。
- **W-E7** 新建对象播种默认键：`createTransition` 写 `priority:0`（磁盘 137 条迁移全无该键）、`createState` 播 `label:id`、备注清空留 `description:''`——对 agent 维护的 JSON 是字节噪音。

## 六、"中型开放世界全部任务与故事落地"胜任度评估

### 撑得住的（架构优势）

1. **组合式并行**：图=独立状态机、信号全局广播、编排只是组织单位——并行故事线、跨线汇聚（reactiveAll/Any）、支线/主线解耦（挂不挂主图监听）都是一等公民。当前 49 图规模下运行时开销可忽略，扫描线性，扩到数百图无性能问题。
2. **进度语义完备**：active + reached 单调历史覆盖"正在/曾经"两类门控；`@owner/@scene` 相对寻址支持模板化内容；quests 纯镜像零双写。
3. **量产管线在位**：模板盖章（{{taskId}}__ 构造性防撞名、三产物全有全无）+ 编辑器三方校验 + validate-data 门 + 本轮验证的全链路审计脚本——批量产任务的工艺链完整。
4. **可测可诊**：真实数据集成测试（全主线推演 + 乱序保护 + 九宫格分支矩阵）、trace/debugSnapshot/dev warp/命令通道——比多数同体量项目的叙事层可观测性好一个档次。
5. **存档演化**：图/状态改名迁移映射机制在位（开放世界长线运营的刚需）。

### 撑不住/要先补的（按优先级）

1. **【必修】存读档原子性（R2/R3）**——开放世界=更长会话+更频繁存读+（大概率要加的）自动存档。这两个红线不修，规模越大踩中概率越高，且踩中即"档废了"级事故。修复成本低（谓词一行 + deserialize 清队列 + epoch），收益极高。
2. **【模型缺口】可重复/实例化任务**：reached 历史**永不复位**、无图重置/运行时实例化原语。日常单（背尸活）靠"每单静态盖一张图"可行，但"无限重复的日常委托/随机事件"没有干净表达——循环回 initial 的图第二轮起 reached 门全部失真。开放世界若需要 repeatable content，要么加"重置子图（清 reached+回 initial）"原语（L2 级新能力，含存档语义设计），要么接受纯 flag 计数器建模（丢掉这套图的表达力）。**这是胜任度问题里最实质的一个，建议先拿设计需求对一次。**
3. **【时间维度】** 无时间驱动迁移；现有 DayManager 延迟事件是 day 粒度、且延迟动作入档情况需单独确认。开放世界的"限时任务/时段事件"要额外设计（zone+flag 拼装可绕，但会退化成 flag 面条）。
4. **【编辑器规模化】** 主编排已 70 元素（30 对话黑盒+34 wrapper），canvas/校验面板/信号目录在 5-10 倍内容量下的可用性未经受考验；已知"每键全量序列化"性能债此前刻意未修。建议在内容翻倍前做一次编辑器压力试产（用模板盖 50 单看手感）。
5. **【纪律工具化】** 顺序反转语义（R1）、广播态里放长时动作（R3 放大器）、任务奖励与叙事演出并发——这三条今天全靠"作者知道"。规模化=作者变多=必须落进校验/机制卡，否则必然有人踩。
6. **【运行时告警口径】** `reportUnlistenedSignal` 不认条件读、只报一次——广播态只被条件叶子消费时的告警噪声已有记忆记录；规模化后这类"狼来了"会稀释真告警，建议把"被条件读"纳入豁免。

### 一句话结论

**骨架能胜任中型开放世界（复杂度线性、语义确定、组合式扩展、工艺链完整），但要先偿还两笔红线级存读档债（R2/R3），并就"可重复任务建模"补一次设计决策；编辑器与纪律工具化是规模化的第二梯队投资。**
