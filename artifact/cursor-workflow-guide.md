# Cursor 工作流使用说明

本文档说明项目内几个 Cursor skill 的职责、适用场景和推荐用法。

## 一、整体思路

这个项目的目标不是让 AI 自由发挥，而是让 AI 在固定流程里工作：

1. 先判断需求属于哪一类
2. 再读取对应文档和相关代码
3. 按架构约束实施
4. 完成后做局部回归 review
5. 定期做全项目 review，控制技术债

当前项目建议主要使用以下 3 个 skill：

- `gameplay-iteration`
- `feature-iteration`
- `core-framework-architecture-review`

---

## 二、Skill 说明

### 1. `gameplay-iteration`

**作用：**

用于处理会影响玩法设计、玩家体验或内容定义的迭代。

**典型场景：**

- 新增或调整任务、支线、主线推进
- 修改遭遇逻辑、规矩逻辑、规矩碎片逻辑
- 修改对话条件、奖励、分支
- 调整物件、货币、商店、奖励
- 修改场景交互、热区、解锁逻辑
- 修改玩家可见的规则和结果

**使用时机：**

当你怀疑这次改动会影响 `玩法功能需求清单.md` 时，就优先使用它。

**它会做什么：**

1. 先判断需求是否真的属于玩法迭代
2. 读取 `玩法功能需求清单.md`
3. 检查是否与现有玩法定义冲突
4. 如果冲突，先报告并停下，等你决定
5. 如果不冲突，先更新玩法文档
6. 再读取相关代码和架构文档实施
7. 完成后做小范围回归 review
8. 如果 review 发现问题，先修复再结束

**不适合的情况：**

- 纯重构
- 架构收口
- 生命周期修复
- 不改变玩法规则的 UI/技术实现细化

**示例提法：**

```text
用 gameplay-iteration 处理这次需求：
新增一个支线任务，要求与当前玩法功能需求清单不冲突。
如果冲突先报告，不要直接实现。
如果不冲突，先更新玩法文档，再实施代码，并在结束前做局部回归 review。
```

---

### 2. `feature-iteration`

**作用：**

用于处理不改变玩法设计前提下的技术迭代、重构和实现细化。

**典型场景：**

- 架构修复、解耦、职责收口
- 生命周期修复
- 存档链路修复
- 事件链、动作链修复
- 性能优化
- UI 表现细化
- 工程配置和工具支持

**使用时机：**

当你的目标是改代码质量、结构、稳定性、表现，但不希望改变玩法设计时使用。

**它会做什么：**

1. 先确认任务不影响玩法
2. 读取 `游戏架构设计文档.md`
3. 读取相关代码和必要的近期 review 文档
4. 用最小必要改动实施
5. 完成后做局部回归 review
6. 若发现影响玩法，停止并建议切换到 `gameplay-iteration`

**不适合的情况：**

- 新玩法
- 改任务设计
- 改遭遇规则
- 改奖励、资源流、玩家结果

**示例提法：**

```text
用 feature-iteration 处理这次需求：
修复 RuleUseUI 的分层问题，不要改变玩法规则。
如果发现会影响玩法设计，先停下来告诉我。
完成后做局部回归 review。
```

---

### 3. `core-framework-architecture-review`

**作用：**

用于做全项目级别的架构审查和技术债盘点。

**典型场景：**

- 近期做了几轮中型迭代，想检查架构是否又长歪了
- 想盘点耦合、分层、生命周期、数据驱动、动作执行问题
- 想检查架构设计文档与代码是否同步
- 想定期清理大技术债

**使用时机：**

建议在以下时机使用：

- 每完成 3 到 5 次中型迭代后
- 涉及 `Game.ts`、存档链路、事件体系、SceneManager、CutsceneManager 的大改后
- 你感觉模块边界开始混乱时

**它会做什么：**

1. 先探索整个项目
2. 对照 `游戏架构设计文档.md`
3. 必要时先同步文档
4. 输出架构问题清单和优先级
5. 默认不修复，等你明确说“开始修复”

**示例提法：**

```text
用 core-framework-architecture-review 做一次全项目 review。
先对照架构设计文档和当前代码输出问题列表，不要直接修复。
```

## 三、怎么选 Skill

### 先问自己一个问题

这次改动会不会改变玩家的规则、结果、资源流、进度流、对话结果或玩法体验？

- 如果会：用 `gameplay-iteration`
- 如果不会：用 `feature-iteration`
- 如果你不是要实现需求，而是要盘点整体问题：用 `core-framework-architecture-review`

### 如果拿不准

可以直接这样问：

```text
先判断这次需求应该使用 gameplay-iteration 还是 feature-iteration。
如果会影响玩法，就按玩法迭代流程走；否则按纯功能迭代流程走。
```

---

## 四、推荐工作流

### 日常迭代

1. 先选 skill
2. 明确本次目标和禁止项
3. 限定改动范围
4. 要求结束前做局部回归 review

### 周期性体检

1. 用 `core-framework-architecture-review` 做一次全项目 review
2. 看高优先级问题清单
3. 你决定是否开始修复
4. 若修复，再按小步修复推进

---

## 五、推荐提示词模板

### 玩法迭代模板

```text
用 gameplay-iteration 处理这次需求。
先检查是否与玩法功能需求清单冲突。
如果冲突，先报告并给出方案，不要直接实现。
如果不冲突，先更新玩法文档，再实施代码。
完成后做局部回归 review，并先修复发现的问题。
```

### 纯功能迭代模板

```text
用 feature-iteration 处理这次需求。
目标是在不改变玩法设计的前提下完成实现。
如果你发现会影响玩法，请先停下来告诉我。
完成后做局部回归 review，并先修复发现的问题。
```

### 全项目 review 模板

```text
用 core-framework-architecture-review 做一次全项目 review。
先输出问题清单和优先级，不要直接修复。
如果需要同步架构文档，可以先同步文档再出报告。
```

---

## 六、补充说明

- 本项目不建议让 AI 一次性处理“大而全”的需求。
- 玩法相关需求优先看文档一致性，不要跳过玩法审查。
- 技术类需求也要防止偷偷影响玩法。
- 全项目 review 不需要每次都做，但必须定期做，否则技术债会继续增长。
