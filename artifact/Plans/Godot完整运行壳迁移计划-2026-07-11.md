## 0. 先把“搬完”说清楚

本任务不是把部分玩法改写成 Godot，也不是把当前 Demo 做一个能走路、能对话的版本。

完成后的模型是：

```text
                         ┌─ TypeScript / Pixi 运行壳
唯一内容源 + 唯一媒体源 ─┤
                         └─ Godot 运行壳
```

两套运行壳共同读取：

- `public/assets/**`：场景、图对话、叙事图、动作、条件、任务、物品、规矩、过场、小游戏配置等。
- `public/resources/runtime/**`：背景、角色动画图集、图片、音频、小游戏媒体等。

Godot 不得出现第二套人工维护的场景、对话、任务或资源定义。Godot 可以有自动生成的缓存和
导出包收集目录，但这些产物必须能从权威源一键重建，且禁止人工编辑。

### 0.1 完成判据

以下条件全部成立才算搬完：

1. 现有全部内容 JSON 都能被 Godot 原样读取，无需增加 Godot 专用内容字段。
2. 101 种 Action、9 类 Condition 节点、7 类图对话节点和 16 类过场呈现步骤全部实现。
3. 26 个生命周期系统、25 个 UI 类、11 个渲染类、3 类世界实体与 4 类小游戏均有等价能力。
4. 相同初始状态和相同命令序列，在两套运行壳中产生等价状态快照、事件序列和存档数据。
5. 27 个场景全部能加载；所有 NPC、热点、区域、场景切换、条件显隐和 onEnter 均可运行。
6. 不使用调试跳转，玩家可从新游戏开始完整走通当前可玩流程。
7. Godot 可在干净环境启动；若要求发行，则可自动收集资源并独立导出。
8. `godot_port/tools/audit_runtime_coverage.py --strict` 通过，且全量回归无未登记例外。

任何“编译通过”“茶馆跑通”“大多数 Action 支持”都不构成完成证据。

## 1. 已确认原则与待确认决策

### 1.1 已确认原则

1. **内容单一真相源**：内容仍只维护在现有 JSON 与 runtime 媒体目录。
2. **TypeScript 是迁移期行为权威**：列举型清单以代码为准，不以架构文档中的旧清单为准。
3. **Godot 是运行壳，不是内容转换工程**：不能把场景 JSON 手工改造成 `.tscn` 后分叉维护。
4. **功能语义原封不动**：条件、动作顺序、失败路径、状态机、存档和生命周期都要对齐。
5. **先契约、后实现**：每个模块先固定输入、输出、事件、存档形状和对账办法，再写 Godot 实现。
6. **不保留当前样机作为架构基础**：现有 `godot_port` 只证明外部资源可以动态读取；正式实施时按本计划重建。

### 1.2 会改变架构的未知项

这些不是实现细节，执行前需要制作人确认。计划先采用括号内默认值：

1. **独立发行包：已确认需要 macOS 与 Windows**（制作人 2026-07-11 签字）。
   - 固定采用“开发态直读 + 构建态自动资源收集”双路径。
   - 导出资源只能由引用图生成，不允许人工维护第二份媒体目录。
2. **存档载荷双向互通：已确认**（制作人 2026-07-11 签字）。
   - 浏览器 `localStorage` 与 Godot 文件槽位的物理介质不同，但 `version/timestamp/systems` JSON 载荷、系统键、版本迁移和失败回滚语义一致。
   - TypeScript 导出的存档可由 Godot 读取；Godot 导出的存档也可回到 TypeScript 继续。
3. **画面一致性口径：已确认肉眼等价，允许跨引擎栅格微差**（制作人 2026-07-11 签字）。
   - 阴影方向、遮挡关系、色调、UI 布局、动画帧、镜头、构图与时序必须一致。
   - 抗锯齿、采样和字体栅格导致的细微像素差异不作为失败。
   - 自动截图差异负责发现问题，最终由 agent 目视裁决，不以单一像素阈值自动放行。
4. **首发平台顺序**：已确认先在 macOS 完成开发与全量验收，再完成 Windows 导出与干净机器验证；Godot Web 不列入首批交付门槛（制作人于 2026-07-11 签字确认）。
5. **实现语言：已确认采用 GDScript + Godot Shader**（制作人于 2026-07-11 签字确认）。
   - 统一使用当前 Godot 4.7 标准版工具链，不引入 Godot .NET 与 .NET SDK 部署依赖。
   - GDScript 动态类型与 TypeScript/JavaScript 语义的差异，必须通过 JSON 契约校验、命令覆盖审计和双运行时差分测试约束，禁止依赖语言默认转换来猜测兼容行为。

## 3. 必须迁移的能力清单

### 3.1 核心层

| TypeScript 权威 | Godot 目标职责 | 关键不变量 |
|---|---|---|
| `Game.ts` | `RuntimeRoot` 组装层 | 只负责依赖注入、启动顺序、tick、存档分发与销毁 |
| `EventBus.ts` | 运行时事件总线 | 系统间不互持业务引用；监听可完整注销 |
| `GameStateController.ts` | 游戏状态控制器 | MainMenu/Exploring/ActionSequence/Dialogue/Encounter/Cutscene/UIOverlay/Minigame 原子切换 |
| `InputManager.ts` | 输入抽象 | 键鼠、触摸、调试注入走同一玩家输入路径 |
| `AssetManager.ts` + `projectPaths.ts` | 资源/JSON 统一入口 | 路径解析、缓存、错误策略、场景预载、销毁语义一致 |
| `StringsProvider.ts` + `resolveText.ts` | 字符串与 `[tag:…]` 展开 | raw 值持久化，显示时 JIT 解析；递归深度与 fallback 一致 |
| `FlagStore.ts` | 唯一 flag 存储 | registry、pattern、migration、值类型、宽松比较、未知 op fail-closed |
| `ActionExecutor.ts` | 统一动作执行器 | 顺序 await、ActionSequence 锁、zone context 线程化、policy 黑名单 |
| `SaveManager.ts` | 三槽存档与回滚 | 非安全状态拒存；坏档先拒；读档失败快照回滚；载荷版本一致 |
| `NarrativeStateManager.ts` | 叙事状态图解释器 | signal 队列、priority、reactive、onExit/onEnter、broadcast、reached、owner 索引 |
| `ScenarioStateManager.ts` | scenario 局部状态 | requires、exposes、manual lifecycle、phase/outcome 与序列化一致 |
| `SceneDepthSystem.ts` | 深度/碰撞配置服务 | depth/collision 同源；floor offset 可临时改和复位 |
| `devRuntimeCommands.ts` | Godot 命令与快照通道 | 与 TS 使用同一命令词汇和快照 schema，供双壳对账 |

```text
    runtime-coverage.md
  generated/        # gitignored；导出态资源镜像/manifest
```

### 4.2 组装与依赖

- `RuntimeRoot` 是唯一跨层组装入口，按 TypeScript `Game` 的顺序创建并注入依赖。
- 系统不直接获取 Autoload 单例；共享服务通过构造/配置注入，保持可测试与可销毁。
- 每个系统遵守统一生命周期接口；`destroy` 必须可重复调用，重 `init` 与首次一致。
- 系统间通信只走 EventBus、FlagStore 和显式注入的窄接口。
- 玩家可见行为只走统一 ActionExecutor；条件只走统一 ConditionEvaluator。

### 4.3 内容加载策略

#### 开发态

Godot 根据 `res://` 的父目录定位仓库根，直接读取 `public/assets` 与 `public/resources/runtime`。
图片、音频和 JSON 走自定义外部加载器，避免 Godot 自动导入 1.2GB 全资源。

#### 导出态

`collect_export_assets.py` 从全部 JSON 引用和权威入口构建资源 manifest，只把实际需要的文件复制到
`godot_port/generated/public`，导出后原样放入 macOS bundle / Windows exe 同包的 `shared/public`。这是构建产物，不是第二内容源：

1. 每次导出前全量重建。
2. 产物带源路径与 hash。
3. 缺引用、越界路径、大小写冲突直接失败。
4. 构建后用同一 ResourceLocator API 读取，不让业务层知道开发/导出差异。

### 4.4 JSON 解释原则

- 不把现有 JSON 批量转换为 `.tres/.tscn` 作为运行真相源。
- 可在内存中规范化成 typed Dictionary/Resource，但必须保留 raw 数据和精确默认语义。
- 未知字段运行时容错但开发态可见；未知动作/条件按现有 fail-closed/fail-loud 口径处理。
- 所有默认值、别名、宽松比较和数字转换必须用对账测试锁定，不凭语言直觉重写。

## 8. 风险与应对

| 风险 | 后果 | 应对 |
|---|---|---|
| Pixi shader 与 Godot CanvasItem shader 坐标系不同 | 深度、阴影、AO 看似实现但脚点/方向错误 | 先锁数学向量和固定点截图，再写 shader；逐场景 A/B |
| GDScript Variant 与 JS 宽松转换不同 | 条件、Action 参数、存档边界行为漂移 | 表驱动差分，不直接使用语言默认等号/转换 |
| Godot 导出不能直接打包工程外文件 | 编辑器能跑、发行包缺资源 | 自动引用图 + generated 镜像；干净机器导出验收 |
| 1.2GB 全目录交给 Godot import | 初次扫描极慢、缓存膨胀 | 开发态外部按需加载；导出态只收集引用资源 |
| 异步动作和叙事信号嵌套 | 顺序错、死锁、旧时间线回写 | 统一 await 队列、epoch/会话 token、动作生命周期压力测试 |
| 迁移期间 TS 权威继续变化 | Godot 追不上或静默缺能力 | coverage 每次从当前源码提取；新增类型立即变红 |
| 字体/富文本/中文回退差异 | UI 尺寸与换行漂移 | 固定字体资产和布局基线；多分辨率截图 |
| 音频格式/解码与 fade 差异 | 首句不同步、环境音残留 | 逐通道音频测试；过场中断/自然结束分别验收 |
| 当前工作区已有大量未提交改动 | 迁移依据与基线可能移动 | 只改 `godot_port` 和本计划范围；每轮记录权威 commit/工作树快照 |
| 当前 Godot 样机存在简化 fallback | 后续误把样机能力当已迁移 | capability 初始归零；执行 P1 时删除并重建样机脚本 |

## 9. 工期管理方式（不做虚假日期承诺）

这是一个约 44K 行权威运行时、1.2GB 媒体和多套自研 shader/小游戏的完整引擎迁移，不能用一个总工期数字
掩盖未知量。执行时按工作包燃尽：

- 每个工作包有明确权威文件、输入 JSON、Godot 交付物和验收命令。

- 每个阶段结束更新 `runtime-coverage.md` 和双壳差异清单。
- 发现语义不清时回到 TypeScript handler/测试取证，不以“Godot 常见做法”填空。
- 任何时候都可以准确回答：总清单多少、已验证多少、还缺哪些、当前阻塞在哪。
