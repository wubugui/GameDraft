## 二、目标模型（v2：电影摄制模型）

**状态 = 永久记录**。叙事状态机是唯一记录本身：每张图此刻在哪个状态、到过哪些状态，**永远入档、永远可查**（条件叶不管该图在不在场都照常求值）。"拍完了"= 图走到终态并永远停在那——不销毁、不搬运、不抄写到任何别的层。

**加载 = 剧组在场（live），卸载 = 剧组收工（dormant）**。live 的包才吃信号、跑反应式、放演出；dormant 的包状态冻结但记录健在。这只是运行行为的开关，与记忆无关。

**导演 = 按清单喊开拍**。到了一个场景（或世界状态变化），导演查清单：哪场戏的条件成立且还没拍完 → 喊开拍（置 live + 起入口演出）；拍完了 → 收工（置 dormant，永久完成的状态留在记录里）。下次进同一场景，可能开拍**另一场**戏——同一场景多行清单、不同条件，天然支持。

## 三、设计

### 3.1 包定义

- 编排（composition）加可选 `package: "<包名>"`——编排本就是内容分组单位，升格为打包单位；无标 = 常驻核心（主线里程碑图等，永远 live）。
- **所有图永远注册、状态永远序列化**——包只影响 live/dormant，不影响注册与存档。

### 3.2 清单（单独文件，裁定③）：`public/assets/data/narrative_packages.json`

每行一场"戏"：

```jsonc
{
  "packages": [
    {
      "id": "听书开场",
      "package": "tingshu_pkg",                 // 指向打了此标的编排组
      "scene": "teahouse",                      // 可选：进此场景时评估（省略=任何状态变化时评估）
      "when": [                                  // 开拍条件：现成条件语言（evaluateGraphCondition 全家福）
        { "not": { "narrative": "scenario_听书", "state": "kicked_out", "reached": true } }
      ],
      "autoPlay": { "type": "startDialogueGraph", "params": { "graphId": "寻狗_听书开场" } },  // 开拍动作（可选）
      "done": [                                  // 收工判据：成立即置 dormant；开拍前置检查 !done 防重拍
        { "narrative": "scenario_听书", "state": "kicked_out", "reached": true }
      ]
    }
  ]
}
```

- `when`/`done` 直接走 evaluateGraphCondition——**不另发明判据词汇**（统一条件源铁律），要查什么状态直接查（裁定④精神）。
- "不再重拍"不需要专门机制：导演开拍前查 `!done`，done 是永存状态的直接查询。

### 3.3 导演（NarrativePackageDirector）

独立小系统，单向看世界与记录：
- 订阅 `scene:enter`（既有中性事件）与 `narrative:stateChanged`；
- 评估清单：`when ∧ ¬done` → 开拍（置 live + 执行 autoPlay）；`done` → 收工（置 dormant）；
- 兜底显式动作 `loadNarrativePackage` / `unloadNarrativePackage`（登记动作四件套），内容可硬调。
- DayManager 保持哑时钟；导演顶多把 day:start 当一个普通触发输入，反向零依赖。

### 3.4 运行时（NarrativeStateManager 扩展，量很小）

- live 集合 = 常驻核心 + live 包（+ 活计激活槽，机制照旧）；`listLiveGraphEntries` 过滤面扩一维。
- dormant 包：冻结语义与活计挂起完全同款（不吃信号/不跑 reactive/状态保留）。
- 条件叶查询**不受 live 影响**——记录永远可读。
- 存档：照旧全量状态 + 新增 `livePackages` 列表（live 是"导演喊过开拍"的历史事实，不能纯靠 when 重推导——存它最稳）。

### 3.5 校验

- 清单引用存在性：package 标必须有编排、scene 必须存在、autoPlay 目标图存在、when/done 走既有条件走查（含 narrativeCount）；
- `done` 恒假结构检测（收工判据引用不可达状态 → 这场戏永不收工）——复用刚做的可达性思路；
- **无跨包读状态限制**（v1 防火墙撤销，裁定④）：任何条件读任何图，天经地义。
- 场景纯化闸：场景 onEnter 出现叙事类动作（startDialogueGraph/emitNarrativeSignal/叙事生命周期动作）→ error（迁移完成后开闸）。

### 3.6 工具

- 模拟器：包 live/dormant 感知 + 导演清单干跑（进场→自动开拍→演→收工），活计感知的直接推广；
- web 编辑器：编排打包标、清单文件编辑页、画布包徽章（live/dormant/已完成三态）；
- 改名级联：清单文件里的图 id/场景 id/编排标入引用登记面（刚做的级联机制直接扩）。

### 3.7 迁移分期

| 期 | 内容 | 量级 |
|---|---|---|
| C1 运行时 | live/dormant 分组 + liveness 过滤扩展 + livePackages 存读档 | **小**（v2 比 v1 缩水：无 unregister 手术、存档基本不动） |
| C2 清单+导演 | narrative_packages.json + Director + 两个显式动作 + 校验 | 中 |
| C3 样板 | teahouse→「听书开场」清单行，onEnter 清空，真机全循环 | 小 |
| C4 全纯化+拆主线 | 其余 3 个梦境场景迁移；xungou_demo_main 按 12 拍拆包（纯内容，可分批） | 大 |
| C5 工具 | 模拟器/编辑器/级联 | 中 |

### 3.8 刻意不做

- DayManager 不动（立碑重申）。
- 活计机器不与包混同：活计=可重复实例（真生灭+计数），包=永存状态的 live 分组。两套并行，互不冒充。
- 不做章节回滚/重播、包并行版本。
- 清单不做优先级/互斥编排（多行同时成立就同时开拍；真踩到再议）。

## 选A重构（2026-07-18，制作人复议"谁触发拍哪场戏"后定案）

**触发的批评（制作人）**：导演"进场 + 没拍过（¬done）就开拍"是错的触发模型——① 听书行漏填 `when`，任何进茶馆都开；② "一场景 100 场戏都绑它、都没拍过 → 一进全开"的灾难可能；③ 更根本：**"拍哪场戏"该由策划触发，用他早就会的图对话 switch，不该由导演发明一套清单去自动枚举**。

**定案（选A）**：
- **进场演哪场戏 = 策划触发**：场景 `onEnter → startDialogueGraph <入口路由图>`，路由图根节点是策划写的 **switch**（按里程碑/scenario 状态选分支或走 done）。引擎不自动决定，只跑策划的图。switch 天生至多走一条分支 → "100 场齐开"不可能。
- **导演退成纯管包水管工**：只按里程碑/场景 `when∧¬done → live` / `done → dormant`，**不触发任何演出**（autoPlay 能力整个拔除，防复发）。包管"谁在听信号"，onEnter 路由管"演哪场"——两层正交。

**改动**：
- 引擎 `NarrativePackageDirector`：删 autoPlay + actionExecutor，只 `setNarrativePackageLive`；Game 构造去 actionExecutor。
- 数据 narrative_packages.json：听书行去 autoPlay（留 scene teahouse + package + done，仍是场景驱动**载包**行）；删 3 个梦 cue 行；剩 14 行**全是包行**（无 autoPlay）。
- 场景 onEnter 恢复：teahouse→`startDialogueGraph 寻狗_听书开场`（该图根 switch 即路由：kicked_out→done / else→fire_cutscene）；梦_夜路/农家院/醒来土路→原 fadeWorldFromBlack+startDialogueGraph（一次性演出，无需 switch）。
- 校验 validator.py：**撤销 C3 的场景 onEnter 纯化闸**（onEnter 现在是进场演出的正当入口）；narrative_packages 行改「必配 package + 禁 autoPlay」（error：进场演出请走 onEnter 路由）。
- 编辑器 narrative_packages_editor.py：**去掉 autoPlay ActionEditor 字段**（详情只剩 id/场景/章节包 + when/done 两折叠块）；tip 明示"导演只管包、戏走 onEnter 路由 switch"。
- 测试：director/flow/editor 三套改为验"纯管包 live/dormant"，autoPlay 相关断言全删。

**这修正了什么**：① 触发权归策划（图对话 switch，planner-native，无新概念）；② "100 场齐开"结构上不可能（switch 单分支）；③ 导演不可能再"自动开戏"（autoPlay 拔除，validator 硬拦）。**包 live/dormant 装卸（C1-C4 的核心价值）原样保留**——headless 已验按里程碑逐拍装卸不变。

**已知语义**：进茶馆仍会经 onEnter → 寻狗_听书开场，其 switch 目前只判 kicked_out；若策划要"某些过场路过茶馆不开听书"，在这张图的 switch 里加条件即可（现在是策划的地盘）。

**验收**：tsc0 / TS 测试（director 纯管包 + DirectorFlow 全主线 + XungouMainFlow）12 绿 / 编辑器 pytest（narrative_packages_editor 6 + construct）18 绿 / validate-data 0err·53warn基线 / 5 文件字节幂等 / PyQt 编辑器离屏渲染确认无 autoPlay 块。全项目门 + 真机（onEnter 触发听书、导演纯管包不触发演出、逐拍装卸、存读档）由 validator + headless 双 agent 复核。

## 包降级为纯组织标签（2026-07-19，制作人复议"包还有什么用"后定案）

**触发的追问（制作人）**：既然"进场演哪场戏"是图对话 switch（执行层）读叙事状态自己决定的、叙事状态机只是被读被写的账本（不驱动演出）——那章节包 live/dormant 到底 gate 了什么？

**证据盘点（结论：包对本 demo 零运行时正确性作用）**：
- **信号面隔离早已足够**：全项目 131 种信号，129 种单图独占；仅 2 种被多图监听——`state:scenario_背尸:fled`（脊椎 + 梦，两者都该响，正当扇出）、`__draft__`（草稿占位，真机不触发）。**零真实串扰**。
- **`processTrigger` 本就按 `from` 态门控** + 信号 id 全限定 `state:<图>:<态>` → 图与图彼此隔离，dormant 只是**冗余的第三层锁**。
- **"按需加载"从未兑现**：所有图启动即 `compileNarrativeGraphs` 编译进内存，包只在信号 scan 时过滤，不省内存/不省加载。
- **性能可忽略**：49 图 / 139 transition，每信号扫一遍微秒级。
- **无一条边依赖 dormant 才正确**：把全部图设为常驻（always-live），demo 行为一模一样（背尸→梦握手照成立、无误触）。

**定案（降级，不是砍）**：保留包作为**纯组织标签**，切断其运行时正确性效果。
- **运行时**：`NarrativeStateManager.listLiveGraphEntries` → 重命名 `listScannableGraphEntries`，**删掉 `if (graph.packageId && !livePackages.has(...)) continue` 这行包过滤**（唯一门控点）。所有已注册图恒吃信号/恒跑 reactive，与包 live/dormant 无关。`applyPackageLifecycle` 去掉 `kickReactiveEvaluation`（切换标记不再产生新反应机会）。**活计实例过滤（挂起冻结）保留**——那是另一台机器。
- **保留但语义降级**：`packageId`（图属哪章）、`livePackages`/`setNarrativePackageLive`（导演维护的"当前活跃章节"集，随存档持久化、供章节感知 UI/工具查询）、导演、清单 narrative_packages.json、章节导演编辑器——全部**只承担组织/分组/工具展示，不 gate 行为**。
- **写死此语义防复发**：validator docstring/消息、`NarrativeStateManager`/`NarrativePackageDirector` 注释、四个测试文件全部改成"包=纯组织标签，不承担运行时正确性"。`NarrativeStateManager.test.ts` 的 dormant 冻结用例**翻转**为"不标活跃也照吃信号推进"；`XungouMainFlowIntegration.test.ts` **删掉"加载全部包"的 workaround**（不加载包照走全主线 = 降级的活证明）。

**远期何时才需重新引入门控**：真 scale 到几百张图、或要加"某章期间的环境反应式行为且共享泛信号（zone/day 那种非全限定信号）会误触发"时。**今天不需要。**

**验收**：tsc0 / 受影响 4 测试文件 76 绿（含翻转用例 + XungouMainFlow 不加载包照过）/ 编辑器 narrative_packages 测试 6 绿 / validate-data 0err（未改任何 JSON，数据面不变，narrative_packages 零 warning）/ validator + headless 双 agent 复核（headless 专验"包标非活跃时 scenario 照样吃信号推进"）。
