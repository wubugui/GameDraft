## 一、现状与接缝(为什么不是从零接)

游戏里已存在三块相关设施,决定了接入姿势:

1. **光环境系统**:`SceneLightEnv`(静态) + `lightEnvCurve.ts`(手调折线:玩家位置
   → 弧长 t → 插值光环境)→ `resolveLightEnv` → shadowField / 实体阴影逐帧消费。
   **接缝就在这**:probe 烘焙是"换生产端"——同样输出 ResolvedLightEnv 语义的量,
   消费端(阴影、着色)不动。手调曲线降级为无烘焙场景的回落项。
2. **深度已在船上**:场景媒体自带 `raw_depth_rg.png`,`DepthOcclusionFilter` 已消费。
   烘焙管线应改为**读写这份正典深度**,不再私养缓存(单一深度源,防漂移)。
3. **坐标桥是现成的**:游戏实体位置(scene worldWidth 单位)↔ 背景像素的映射
   SceneManager 已有;实验室 work-res(512宽)只是同一张图的缩放。
   桥 = 实体世界坐标 → 背景像素 → ×(512/native) → 地面深度场 d_walk → q → M → 伪世界
   → probe 三线性。全程每帧每实体几十次乘加,CPU 白给。

## 二、目标形态(一句话)

> 每个进入场景的角色/NPC,每帧从该场景的 **L2 probe 烘焙**里取一份辐照度,
> 以 **整体 tint(L0)** 起步、**片元滤镜(带法线)** 进阶,替代手调光环境曲线;
> 无烘焙的场景自动回落旧曲线,玩家无感切换。

## 三、运行时载荷(每场景,随场景媒体加载)

放 `public/resources/runtime/scenes/<id>/lighting/`,由实验室新增 `--export-runtime` 产出:

| 文件 | 内容 | 尺寸估计 |
|---|---|---|
| `lighting.json` | 标定(M/ppu/cx/cy/theta)+ probe 网格 + ambient SH + **背景内容哈希** + 版本号 | ~4KB |
| `probes_l2.bin` | (P,9,4) f16:E_hit rgb + 覆盖率(维持三件套语义,运行时闭包可调) | ~120KB |
| `probes_l2amb.bin` | miss 环境拆分项 | ~90KB |
| `ground_d.png` | d_walk 地面深度场,RG16 编码,work-res | ~300KB |

合计 ~0.5MB/场景,30 场景 ~15MB(走 DVC/OSS 同现有媒体)。
**不进游戏**:体素、mesh、点云、BIN/L1 cache、walkable(游戏有自己的 collision)、RT。

**防腐门(A2 教训,必须做)**:`lighting.json` 携带 background.png 内容哈希;
运行时不符 → 禁用烘焙 + console warn;validator 升 error。杜绝"背景重画烘焙静默过期"。

## 四、运行时系统

新建 `src/systems/CharacterLightingSystem.ts`,实现 IGameSystem,构造函数注入,
EventBus 听 `scene:changed` 载/卸载荷(§1 八条对照:分层✓ 解耦✓ DI✓ 数据驱动✓
统一生命周期✓;serialize/deserialize 无状态直通)。

对外只有一个查询面:
```ts
sampleE(worldX, worldY, opts?) → {rgb: E, cov} | null   // null = 本场景无烘焙
```
内部:坐标桥 + 8 probe 三线性 + 闭包(E_hit + s·E_amb,s 沿用实验默认)。

**消费端两阶段**:
- **阶段A·tint(先上线)**:每实体每帧 1 次 sampleE(胸口高度)→ 折算成 tint 色乘到
  sprite(显示定标:场景锚点处 tint≈白,只表达相对偏移——A1 的补救,防"角色变泥")。
  覆盖玩家 + 全部 NPC,零 shader 工作量。
- **阶段B·片元滤镜(验收后)**:玩家(可选主要 NPC)换 Pixi Filter,片元级
  probe 插值 + 隆起法线(char/normal.png 管线化),身体出现受光梯度。
  实验室 CHAR_FS 裁剪移植,只保留 L2 路径。

**与旧系统的优先级**:场景有烘焙 → 本系统接管 tint/光环境语义;无 → lightEnvCurve
原样生效。禁止两者同时乘(双重记账,同 A8 纪律)。

## 五、内容工作流(编辑器往返红线)

- 烘焙参数(pitch/ppu/relief/EV/max_gain_ev/fold/probe 网格)= 内容,存
  `lighting.json` 的 params 块;**实验室查看器就是这个域的编辑器**
  (同 scene_depth_editor 先例):调参 → 页内重建 → `--export-runtime`。
- 策划流程:新场景背景定稿 → `./dev.sh char-lighting -- --build ... --export-runtime`
  → 进游戏即生效。默认参数已可全自动,调参是可选项。
- validator 新增三查:载荷齐全性 / 哈希一致 / json schema。素材审计登记新文件类型。

## 七、风险与明确不做

**风险**:①与透视缩放系统的叠加顺序(tint 与 scale 无冲突,滤镜阶段注意 filter 链顺序);
②尺度感不匹配(ppu 是拍板参数,试点期用查看器现场校);③31 场景批烘的个别翻车
(标定失败的场景先回落曲线,不阻塞)。

**明确不做(本轮)**:游戏内 RT/BIN/L1(实验室专用)、SH 升阶、mesh 进游戏、
实验室碰撞替换游戏碰撞、3D 检视进编辑器、每帧动态重烘。
