# 叙事运行实例化（可重复运行的活计机器）— 技术设计 v2（2026-07-17）

> 玩法需求：`docs/玩法功能需求清单.md` D5（多接单·单激活·可重复刷新）。
> 前置：`artifact/Reviews/叙事状态机全面审查-2026-07-17.md`（R2/R3 已修是地基——reset/revert/挂起依赖"时间线失效/基线复位"语义）。
> **v2 修正（制作人复议 2026-07-17）**：v1 曾设计"并发多实例（图@key）"，被否。定案：**运行时永不并存两台一样的叙事**；实例是"一台可反复从头开始、可挂起恢复的机器"，不是"可复制的机器"。挂起/恢复/重开/回退是状态机分内功能，按图配置。本文即 v2 契约，v1 的 @key 多实例/量词叶/实例形态广播全部作废。

## 0. 定案要点（制作人拍板）

1. **运行时不并存两台一样的叙事**：每个活计原型至多一个实例。
2. **激活只有一单**：全局单激活槽——同一时刻只有"激活"的那个活计实例在收信号推进；其余活计实例冻结（挂起）。常驻图不受此限，永远都在跑。
3. **挂起/重开按图配置**：`resumable` 决定切走时挂起（存快照续玩）还是弃置（切回从头）。
4. **"接很多单"是任务层的事，不是引擎的事**：引擎只管"正在跑的这一次"+累计计数；接了几单、面板怎么列，是 S2 QuestManager/HUD 的簿记，与状态机无关。
5. 节奏：S1 引擎 → S2 编辑器+任务面板 → S3 绑定+调度，期间背尸单真机验收。

## 1. 概念模型

- **图两类**：**常驻图**（无 `run` 声明；现有 49 张，永远一台机器在跑，语义分毫不变）+ **活计图**（有 `run` 声明；可反复运行的机器）。
- **活计图 = 每原型至多一个实例**。因为不并发，**实例 id 就是 graphId 本身**（无 `@key`）——运行时表结构复用现有 `activeStates`/`reachedStates`，活计图的条目在"弃置/结算"时删除、下次开跑重建，常驻图的永不删。这是 v2 相对 v1 的核心简化。
- **全局单激活槽** `activatedArchetype`：只有它指向的活计图参与信号/reactive 推进；其余有实例的活计图=**挂起**（状态在表里冻结，不推进）。
- **历史两层**：实例内（active + reached，随实例生灭）→ 跨轮累计（原型级 counters：started/reset/aborted/settled-by-exit，随存档持久化，是"做过几次"的唯一真相）。reached 不跨轮（随实例删）——跨轮门控一律用 counters。

## 2. 数据契约

### 2.1 图声明（narrative_graphs.json）

```jsonc
{
  "id": "run_背尸_淹尸",
  "ownerType": "scenario",
  "run": {                    // 缺省 = 常驻图（兼容全部存量）
    "repeatable": true,        // 走到出口结算后可再次开跑（计数 +1）
    "resumable": true          // 切走时挂起存快照续玩；false=切走即弃、切回从头
  },
  "initialState": "accepted",
  "entryState": "accepted",
  "exitStates": ["delivered", "failed"],   // 到达任一出口 = 结算
  ...
}
```

### 2.2 存档 v2

```jsonc
{
  "version": 2,
  "singletons": { "<graphId>": { "active": "s", "reached": ["..."] } },
  "runs": { "<archetypeId>": { "active": "s", "reached": ["..."] } },   // 每原型至多一条
  "counters": { "<archetypeId>": { "started": 3, "reset": 1, "aborted": 0, "settled": { "delivered": 2 } } },
  "activatedArchetype": "run_背尸_淹尸",   // 或 null
  "activeStates": {...}, "reachedStates": {...}   // v1 兼容字段（仅常驻图）
}
```

v1 旧档（扁平 activeStates/reachedStates）读入 → 整体升格为 singletons，runs/counters 空、激活槽 null（= 活计功能出现前的语义）。migrations（改名映射）作用于 singletons + runs 两层。存档仍受 `isIdle()` 门（R3）约束，生命周期都在排空内。

## 3. 运行时（NarrativeStateManager）

### 3.1 表

复用 `activeStates: Map<graphId, state>` / `reachedStates: Map<graphId, Set>`（常驻图恒在；活计图有实例时在、无则不在）。新增：
- `activatedArchetype: string | null` —— 当前推进的活计图。
- `suspendedRunArchetypes: Set<string>` —— 有实例但非激活（挂起冻结）。
- `runCounters: Map<archetypeId, RunCounters>`。

**信号/reactive 迭代单位** = 全部常驻图 + `activatedArchetype`（若非空）那一张活计图。挂起活计图冻结、不参与。（v1 的 `listLiveInstanceEntries` 简化为此。）

### 3.2 生命周期（四动作，全经队列，与 generation 代际/isIdle 同时序）

| 动作 | 语义 | 已有实例时 | 无实例时 |
|---|---|---|---|
| `startNarrativeRun(A)` | 开一轮新的并激活 | 警告跳过（重来用 reset、切回用 activate） | 建 initialState 实例、激活 A、挂起/弃置前一激活 |
| `activateNarrativeRun(A)` | 切换激活到已存在实例（恢复挂起） | 激活 A、挂起/弃置前一激活 | 警告跳过（开新用 start） |
| `activateNarrativeRun('')` | 清激活槽 | 当前激活的按 resumable 挂起/弃 | — |
| `resetNarrativeRun(A)` | 当前实例回 initialState + reached 清空（"从头再来"，静默重基线，不碰激活槽） | 计数 reset++ | 警告跳过 |
| `revertNarrativeRun(A, s)` | 当前实例回退到状态 s（静默重基线；改 active，**不抹 reached** 历史，不碰激活槽） | s 须存在于图 | 警告跳过 |

**挂起 vs 弃置**（activate/start 切走旧激活时）：旧激活图 `resumable:true` → 记录留在 runs 表、进 suspendedRunArchetypes（冻结续玩）；`resumable:false` → 删除实例（active/reached 出表）+ 计数 aborted++。

**结算**（激活实例到 exitStates，enterState 内自动）：counters.settled[exit]++、删实例、清激活槽。做完总是删（不留挂起）；下次接=start 新一轮。

### 3.3 静默重基线（reset/revert 的硬语义）

reset/revert 跟读档同口径：**不跑沿途 onExit/onEnter、不重放广播**。理由是 R2 那类雷——revert 回中间点若重发落点广播，监听它的图被"重复历史"再推一次=时间线污染。重置/回退是抹除，不是演出。reset/revert 后 kick 一轮 reactive 重评（内部管线）。

### 3.4 事件（系统层通知，非叙事信号）

`narrative:stateChanged` payload 加 `cause: 'transition'|'reset'|'revert'|'resume'|'settle'|'discard'`（缺省 transition）。PlaneReconciler/QuestManager/HUD 按需过滤 cause；**引擎绝不为生命周期自动发叙事信号**——内容要对"重开/撂挑子"有反应，由触发点自己发普通作者信号（见 §6）。另发轻量事件 `narrative:runActivated {archetypeId, previous}` 供 HUD 追踪切换。

### 3.5 条件叶（6 → 7 类）

单活模型下 **`{narrative: 活计图id, state}` 普通叶直接读活计实例当前态**（有实例命中、无实例 false）——v1 的量词叶（any/all/activated）**整个删除**。新增仅一条：
```jsonc
{ "narrativeCount": "<archetypeId>", "exitState": "<stateId>"?, "op": "==|>=|<=|>|<|!=", "value": n }  // 跨轮累计结算次数
```
唤醒源：narrative 叶随迁移/reset/revert/settle 变（全在排空内，排空尾 reactive 重评覆盖）；narrativeCount 随 settle 变（同）；激活切换额外 kick 一次。无沉睡窗口。

### 3.6 广播回归普通形态

活计图 broadcastOnEnter 发 `state:<graphId>:<state>`（普通键，因 graphId 唯一）。但活计重复运行每轮都发，**常驻图/主线裸听活计图广播 = 校验 warning**（提示改用 narrativeCount 条件门）。

### 3.7 其它

- PlaneReconciler 改用"常驻图 + 激活活计图"的激活态派生（挂起活计不点名位面）；活计图 activePlane S1 报 warning（S2 放开）。
- `@owner/@scene` 只解析常驻单例 wrapper；活计图不进 owner 索引（校验拦）。
- setState/debugSetNarrativeState 对活计图拒绝（会造幽灵条目）；dev warp 只作用常驻图。
- debugSnapshot 增列 runs/counters/activatedArchetype/suspended。

## 4. 动作四件套（S1）

`startNarrativeRun` / `resetNarrativeRun` / `revertNarrativeRun` / `activateNarrativeRun`。每条全登记面（ActionRegistry + actionParamManifest + action_editor ACTION_TYPES/_PARAM_SCHEMAS/ACTION_PERSISTENCE/graphId 选择器 + validator + CONTENT_ID_PARAMS + _SELECTOR_KIND_UNIVERSE + project_model id-provider）。graphId 走 `narrative_run_archetype` 选择器（候选=声明 run 的图）。revert 的 stateId 走该图状态选择器（S2；S1 JSON）。**结算不设动作**（exitStates 自动）。

## 5. 校验防火墙（TS 权威 + Python 子集 + validator.py 三面）

1. `run` 声明形状（repeatable/resumable 布尔）；活计图须有 entryState+exitStates（复用 scenario 规则）。
2. start/reset/revert/activateNarrativeRun 的 graphId 必须指向活计图（error）；revert.stateId 须存在于该图。
3. `narrativeCount` 指向非活计图 = error；exitState 须是该图出口。
4. **常驻图/主线裸听活计图 `state:图:态` 广播 = warning**（每轮重发；改用 narrativeCount）。
5. 活计图不得作 wrapper owner（error）；activePlane = warning（S1）。
6. id 禁 `@`/`:`/`|`（`@` 是相对 token 保留字；现网 0 处）。

**v1 已删规则**（单活后合法）：`narrative` 叶指活计图不再是 error（普通叶直接读）；实例形态广播 error、量词校验 —— 随 @key/量词叶一并删除。

## 6. 生命周期的信号策略（钉死）

| 事 | EventBus | 自动叙事信号 | 内容要反应 |
|---|---|---|---|
| reset/revert/resume/settle/discard | `narrative:stateChanged` + cause | **无** | 触发点自己发作者信号（如失败结算点发 `beishi_job_failed`） |
| 面板切走弃置 | 同上（cause: discard） | 无 | 条件表达（无活 run + 接单计数>完成计数）；将来真需要再加可选声明 `onDiscardSignal`（作者命名、进目录、非保留键） |

信号目录里永远只有作者写的语义信号；引擎生命周期动静全在事件层。settle 同理：cause:'settle' 事件 + 计数器++，主线联动用 `narrativeCount` 条件门、不听事件——进度真相源永远是状态和计数。
