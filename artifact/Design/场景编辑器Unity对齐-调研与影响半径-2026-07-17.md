## 一、现状快照（关键事实）

### 数据模型（src/data/types.ts）
- 场景只有 4 类可放置实体，全部**扁平数组/字典**存储：`hotspots[]`(HotspotDef :250)、
  `npcs[]`(NpcDef :619)、`zones[]`(ZoneDef :1447)、`spawnPoint(s)`(:189-190)；另有 `backgrounds[]`(:782)。
- **没有独立的 item/装饰实体类型**：物件=hotspot.displayImage；装饰烤入实体=NpcDef.renderRaw:true 的伪 NPC。
- 空间字段现状：只有世界坐标 `x/y`（zone 是 `polygon[].x/y`，patrol 是 `route[].x/y`）。
  **全库不存在任何实例级 scale/rotation/transform/parentId 字段**（scenes/*.json 全量确认）。
- 唯一"近亲"是朝向镜像：`initialFacing`/`displayImage.facing` → `sprite.scale.x = ±1`。

### 运行时渲染（Pixi.js v8）
- 层树：`worldContainer` → `backgroundLayer/shadowLayer/entityLayer`（Renderer.ts:36-76），
  每个实体是独立 `PIXI.Container` **扁平直挂 entityLayer**（SceneManager.ts:970-1010）。
- rotation 全库无人用；scale 只用于镜像与"帧像素→世界尺寸"映射（SpriteEntity.ts:365-378）。
- **`Npc.setFacing` 取 `Math.abs(container.scale.x)` 作 base 再乘符号（Npc.ts:204-207）——
  引入实例 scale 后会被镜像吞掉，是必修点。**

### 坐标消费方分裂（本次迭代最大隐患）
两套读法今天恰好一致（container.x/y === def.x/y），加 transform / 父子后立即分裂：

| 读 **def 原始坐标** | 读 **显示对象坐标（假定=世界）** |
|---|---|
| 交互命中 InteractionSystem.ts:164-206 | 深度排序 Renderer.sortEntityLayer:188-211（读 child.y） |
| 玩家-实体碰撞 Game.ts:2030-2036（collisionPolygonLocal 只平移不缩放不旋转） | 深度遮挡 Game.ts:4193-4219（遍历 entityLayer.children 读 c.x/c.y） |
| zone 判定 ZoneSystem.ts:131-166 | 逐实体光照脚点采样 SceneDepthSystem.ts:342-361 |
| moveEntityTo/faceEntity ActionRegistry.ts:1381-1437 | 投影阴影 Game.ts:2265-2329 |
| — | 气泡 EmoteBubbleManager.ts:121-129（**硬假定 parent===entityLayer**） |

### 编辑器（tools/editor/editors/scene_editor.py，9402 行单文件）
- 画布=QGraphicsView/QGraphicsScene；拖动走 `_DraggableCircle` → `item_moved` →
  `_on_item_moved`(:8579) → staging → `mark_dirty("scene", sid)`(:8666)。脏桶按场景增量写盘。
- 属性区 `ScenePropertyPanel` 已按类型分化 6 个面板（scene/hotspot/npc/zone/spawn/empty）——
  **Inspector 的"分型属性面板"一半已存在**；缺的是实体层级树 + 搜索选择聚焦入口
  （审查 2026-07-14 明确记为结构缺口："场景内无实体侧栏清单"）。
- 聚焦链路现成：`select_by_id`(:9398) → `_select_scene_entity_by_kind`(:9333) →
  `_focus_canvas_on_entity`(:9357 居中) + `flash_canvas_item`(:9370 闪烁描边)，
  全局搜索(Ctrl+Shift+F, json_lang LSP) 也走这条契约。

### 撤销设施盘点（全项目 5 套，无一覆盖场景画布）
1. **全局 `ProjectModel.undo_stack` 是死栈**：`push_edit` 全库零调用者，Edit→Undo 菜单挂空
   （project_model.py:49,1011；main_window.py:355）。
2. entity_refactor journal（cap 20，单栈无 redo）：只覆盖重构菜单四操作 + Ctrl+D 复制；
   **普通拖动、属性 Apply、Delete 键删除(:9279)、右键新增全都不可撤销**。
3. TimelineEditor：结构操作前整树深快照栈（timeline_editor.py:2771-2832），局部 Ctrl+Z。
4. **图对话编辑器（PySide/OdenGraphQt）：规范 QUndoStack + 快照命令 + mergeWith 合并 +
   首次 redo 跳过（editor_widget.py:93-154,241）——给场景编辑器做 undo 的最佳蓝本**。
5. 叙事 Web 编辑器 useEditorHistory（TS，不可复用到 Qt）。

### 存档
- 实体 x/y/enabled 等已入档（`entityOverrides`，types.ts:748-775；SceneManager.serialize:1708）。
- 可持久化运行时字段由 `public/assets/data/runtime_field_schema.json` 白名单管控，
  TS 侧 `EntityRuntimeFieldSchema.ts:124-163` **硬编码只认已知键**——新字段要运行时可变必须三处登记。
- `SAVE_VERSION=1`，无正式迁移函数（SaveManager.ts:7,96-99）。

## 三、两个选型级红线（调研结论，强烈建议按此定案）

1. **父子数据形态：`parentId` 引用 + 保持 npcs/hotspots/zones 扁平数组。**
   若改成 children 内联嵌套：validator 去重、entity_refactor 扫描、json_lang id_universes
   （id_universes.py:160-212）三处都只遍历顶层数组，会系统性漏掉嵌套子实体，影响半径成倍放大。
2. **运行时父子：逻辑层级，不做 Pixi 真 re-parent。**
   保持实体容器扁平直挂 entityLayer，每帧按拓扑序求解 子世界变换 = 父世界变换 ∘ 子局部变换 后写回容器。
   绕开 visible/scale/destroy 树传播三族冲突（隐坑 5）与"直接子=世界"假设（隐坑 2/6），
   深度排序/遮挡/光照代码可继续读容器坐标（它们拿到的仍是世界值）。代价：新增每帧层级求解器（实体数量级小，可忽略）。

## 四、核心语义决策（待用户拍板，附推荐）

| # | 决策点 | 推荐 | 理由 |
|---|---|---|---|
| D1 | 有 parentId 时 x/y 是局部还是世界 | **局部**（无 parentId 时不变，仍是世界） | Unity 语义；父动子随。代价：moveEntityTo/persistNpcAt/setSceneEntityPosition 等世界坐标 action 在应用点做世界→局部换算（ActionRegistry.ts:876,1097,1115,1381）；重构引擎跨场景迁移/重父时做坐标换算；编辑器数值框显示局部+只读世界 |
| D2 | 缩放/旋转是否影响命中与碰撞 | **影响**（collisionPolygon/interactionRange 随 transform 变换） | 否则"看得见摸不着"；分两步落地可先视觉后命中，但最终必须一致 |
| D3 | transform 覆盖哪些类型 | npc/hotspot 全量（平移已有+缩放+旋转）；zone 编辑器给多边形整体缩放/旋转工具（烘焙进点列，运行时不加 zone transform）；spawnPoint 仅平移；backgrounds 暂不动 | zone 运行时判定读原始 polygon，烘焙式改动零运行时成本 |
| D4 | transform 是否运行时可变（新 action setEntityTransform）并入档 | **首期不做**，静态 def 字段起步；预留登记路径 | 避免首期就动 runtime_field_schema 三处白名单 + SAVE_VERSION；后续按 L2 三件套补 |
| D5 | 撤销范围 | SceneEditor 局部 QUndoStack（图对话命令模式），主菜单 Edit→Undo 改为转发当前激活编辑器的栈 | 全局死栈粒度不适用；转发能顺手救活主菜单 |

## 五、分期实施方案（每期独立可验收，字段缺省时行为与现在完全一致）

**P1 撤销/重做（纯编辑器，先做——后续所有新操作都要长在栈上）**
SceneEditor 内建 QUndoStack；把 `_on_item_moved:8579`、`_apply_props:8944`、`_delete_selected:9279`、
`_add_*_at:9050-9131` 改造为命令（source 前后快照，在 commit 之后建命令，避免与迟到 staging 互踩）；
mergeWith 合并连续拖动；桥接 entity_refactor journal；Ctrl+Z 用 WidgetWithChildrenShortcut；
流程探针（编辑→undo→断言模型）。

**P2 Hierarchy 树 + Inspector（纯编辑器 UI，复用度最高）**
左栏加实体树（先按类型分组的扁平树）+ 过滤框；选中回调接 `_select_scene_entity_by_kind`；
搜索聚焦复用 select_by_id/_focus_canvas_on_entity/flash 链路；树上改 id 走重构引擎（禁裸改口子）。

**P3 实例 transform（运行时 + 编辑器 gizmo，半径最大）**
types.ts 加可选字段 → SpriteEntity/Npc/Hotspot container 级应用（修 facing 吞 scale）→
统一世界坐标解析器，逐个改造隐坑 1-4/6 的消费方 → 编辑器 gizmo + 面板数值框 + validator 护栏 →
真机验证（命令通道）光影/命中/深度一致。

**P4 父子层级（依赖 P3）**
parentId + 扁平数组 + 逻辑层级求解器（拓扑序、防环）→ 树上拖拽重父（世界坐标保持不变的重父换算）→
entity_refactor 四路径登记 parentId + parity 测试 → validator 新增存在性/成环/跨场景检查 →
存档语义评审（如 x/y 局部化涉及旧档，走 SAVE_VERSION 迁移）。

**P5 收尾与严格审查（用户要求）**
验证门全套：编辑器 pytest 全量 + 素材审计 --strict + validate-data（0 error 且 warning 不增）+
`npx tsc --noEmit` + `npm test` + 黄金往返/字节级验收 + 真实 QMouseEvent 流程探针；
然后启动多路子 agent 对全部改动做对抗式严格 review，修完再收。

## 七、轻量方案 B（2026-07-17 二轮定向，首选）

### 需求重述（用户澄清的原始诉求）
1. **复用**：同一素材多次摆进场景，每个实例可移动/缩放/旋转；
2. **运行时可改 transform** 以增强表现；
3. "父子"的真实用途 = **组操作**：对一组实体同时触发同一操作（同时 active/deactivate、同时移动等），
   不需要 Unity 场景树；
4. undo/redo 与 Inspector 需求不变。

### B1 实例 transform —— quad 级真变换（2026-07-17 三轮修订：用户否决"只变表现"）

> 语义拍板：transform 对 quad 呈现的 sprite 实体（npc 的 animFile 帧 / hotspot 的 displayImage）
> **直接变换整个 quad**——凡从 quad 派生的空间量（碰撞、交互、阴影、遮挡）一律跟随，
> 运行时改 transform 同样即时跟随。"视觉-only + 编辑器烘焙"方案被否
> （烘焙=运行时改 scale 碰撞不动，违背诉求）。

**数据**：NpcDef / HotspotDef 加可选 `scale?: number`（等比）与 `rotation?: number`；缺省不写。

**锚点=现有脚底中点（anchor 0.5,1），x/y 恒为世界坐标（锚点不动）**。锚点不变保住的只是
**平移语义类**消费方：相机、zone 判定（玩家点 vs zone 多边形）、patrol/moveEntityTo/faceEntity
（都操作锚点世界坐标）、存档 x/y 含义——这些零改动。
**凡消费实体"范围（extent）"的系统全部受 transform 影响**（四轮修订：此前"深度/光照/气泡零改动"
的说法错误——缩放改变高度与占地，旋转改变接地线/身体中心/头顶位置）。

**核心设计物：变换后 quad 解析器（TransformedQuad，运行时与编辑器同源同口径）。**
输入 def + 当前实例 transform，输出：变换后四角/AABB、接地线 y（sort 用底边 max-y）、
头顶中点（气泡用）、身体中点与受 scale 缩放的探针高度（光照用）、世界尺寸（阴影用）、
变换后碰撞/遮挡多边形、缩放后交互半径。**所有 quad 派生消费方一律改读解析器**，
不再各自摸 container/def 字段——这是 P3 的第一交付物。

**必改的 quad 派生消费方（transform 的真实半径）**：
- 渲染：container 级施加 scale/rotation；修 `Npc.setFacing` 吞 scale（Npc.ts:204-207，
  改 实例scale×facing符号 复合）、Hotspot displaySprite 同理（Hotspot.ts:147-150）。
- **深度排序**：`Renderer.sortEntityLayer`(:188-211) 改读解析器的接地线 y（变换后 quad 底边）。
  纯缩放时底边 y 恰=锚点 y（底中锚），行为与现在一致；旋转时按躺倒后的真实接地线排序。
- **深度遮挡采样**：`Game.ts:4193-4219` 的脚点采样与 `entityOcclusionPolygon` 改用解析器的
  变换后脚点/多边形。
- **逐实体光照**：probe 点改按解析器计算——采样高度 sampleLift 随 |scale| 缩放、
  方向随 rotation（大个子的身体中心更高、躺倒后中心在侧向），保持"脚点锚与深度图同源"约定。
- **气泡**：`EmoteBubbleManager.ts:121-129` 改读解析器的头顶中点（缩放后更高、旋转后偏移）。
- **阴影**：宽高经 `getWorldSize` 计入实例 scale；real 模式剪影跟随显示 transform，
  椭圆模式仅随 scale（rotation 对贴地椭圆的影响首期忽略）。
- **碰撞多边形**：`hotspotCollisionPolygonToWorld` / `npcCollisionPolygonToWorld`
  （hotspotCollision.ts:15-40）当前只平移——扩展为绕锚点 缩放×旋转×平移 的完整变换，
  **求值时读当前实例 transform**（数据仍按未变换姿态编写），运行时改了碰撞自动跟随。
  world 系与 local 系两种 authored 多边形统一绕 (x,y) 锚变换。
- **交互**：`interactionRange` 乘 |scale|（InteractionSystem 取值处一处改）；中心点=锚点不动。
- **编辑器画布**：quad 外框、碰撞多边形叠显、点选命中区域都按解析器几何绘制/判定
  （画布与运行时同口径，防"预览撒谎"）。

**运行时可改 + 入档：不新增 action。** 复用现有 `setEntityField` 通道——把 scale/rotation 登记进
`runtime_field_schema.json` + `EntityRuntimeFieldSchema.ts` apply 分支 + Override 类型三处，
即获得"action 可改 + 存档自动持久化"。因碰撞/交互是**求值时变换**，setEntityField 改完
下一帧命中即正确，无需额外失效逻辑。平滑过渡（tween）不做，后续有需要再按 L2 扩展。

**编辑器**：画布 gizmo（旋转/缩放手柄，参考 `_paint_light_gizmo` 自绘）+ 新 `item_rotated/item_scaled`
信号照 `item_moved` 骨架写回；分型面板加数值框（`_keep_num` 保真）。复用故事由
既有 Ctrl+D 复制（entity_refactor.duplicate）+ 实例 transform 组合完成。
zone 不是 quad，不加 transform 字段——编辑器提供多边形整体缩放/旋转工具（点列烘焙，运行时零改）。

### B2 标签式分组 —— 取代父子层级（radius 大减的关键）

**数据**：实体加可选 `group?: string`（**纯标签，不是 id 引用**）。由此免掉的整族改动：
- 无 id 引用 → entity_refactor **不需要**登记新引用类（改名/删除实体，标签天然无悬垂）；
- x/y 永远是世界坐标 → **D1 作废**：无局部/世界语义分裂、无 action 换算、无 SAVE_VERSION 迁移；
- 无层级 → 无逻辑层级求解器、无拓扑排序、无防环校验、Pixi 树传播冲突族整体消失（隐坑 5/7 消解）。

**运行时**：新增 2 个组动作（各走 add-game-action 四件套登记）：
- `setGroupEnabled { group, enabled }`：遍历组成员走现有**会话显隐通道**（sessionEnabledOverride），
  与显隐四通道模型天然兼容；
- `moveGroupBy { group, dx, dy, speed? }`：成员各自 世界坐标+delta，复用 moveEntityTo 移动机制。
首发覆盖 npc/hotspot（显隐/移动通道现成）；zone 是否入组待验证（其启用通道形态不同）。

**编辑器**：组指派用选择器（既有组名候选 + 新建）；Inspector 树按「类型/组 → 成员」组织
（分组树与 Inspector 树是同一个控件）；画布多选/框选 + 整组拖动（编辑器侧批量改 x/y，整体进 undo 栈）。

**校验**：validator 对组动作引用的 group 做"场景内存在性"检查（warning 级）；
json_lang 组名 universe 作增强项。

### B3 实体列表 + 多选（2026-07-17 五轮补充需求）

**实体列表（P2 树面板的明确职责）**：场景内全部实体（npc/hotspot/zone/spawn）入列表/树
（类型/组→成员），带过滤框；**列表↔画布选中双向同步**（列表点选→画布聚焦闪烁，
画布点选→列表高亮定位）；填补审查已记录的结构缺口"场景内无实体侧栏清单，发现实体只靠画布点选"。

**多选（新增范围，现有选择模型是单选）**：
- 入口：列表 Ctrl/Shift 多选；画布框选（QGraphicsView RubberBandDrag）+ Shift 加选。
- 多选下的批量操作（每个批量操作=**单条 undo 命令**）：整体拖动（成员各写 x/y）、
  Delete 批量删除、Ctrl+D 批量复制、批量指派 group。
- 多选时属性面板显示"已选 N 个实体"+ 批量操作条；**字段级批量编辑（同时改多个实体的
  同名字段）首期不做**，后续有真需求再扩。
- 与既有机制的协调点：staging/commit-on-leave 以"当前编辑实体"为轴（单数假设），
  多选批量操作绕过表单 staging、直接按 source 前后快照走 undo 命令（与 P1 的命令建法同源）；
  重构菜单（迁移/改名）保持单选语义，多选时置灰。

| | 方案 A（Unity 场景树） | 方案 B（轻量） |
|---|---|---|
| 坐标语义 | x/y 局部/世界分裂，action 换算，存档迁移 | 不变（全世界坐标） |
| 重构引擎 | parentId 四路径登记 + parity | 零改动 |
| 运行时结构 | 逻辑层级求解器/拓扑序/防环 | 一个 TransformedQuad 解析器；无层级/拓扑/防环 |
| 空间消费方 | 局部/世界坐标语义分裂，全消费方改造 | 坐标语义不变；extent 类消费方（深度/光照/气泡/阴影/碰撞/交互/遮挡）统一改读解析器；平移类（相机/zone/patrol/moveEntityTo）零改动 |
| 新能力登记 | setEntityTransform 等 | schema 登记 2 字段 + 2 个组动作四件套 |
| 满足诉求 | 超集（含真层级变换继承） | 恰好覆盖复用/运行时表现/组操作 |
