# 渝都卫（FindingDog）— 游戏架构设计文档

## 一、技术栈

- **语言**：TypeScript（严格模式）
- **渲染**：PixiJS 8
- **音频**：Howler.js
- **叙事脚本**：inkjs（ink语言的JS运行时）
- **构建**：Vite
- **桌面打包**：Tauri（后期）
- **数据格式**：JSON（游戏数据）、ink（对话脚本）

---

## 二、架构总览

采用**分层架构 + 事件驱动通信**。各系统模块化，通过中央事件总线解耦，通过全局标记系统（FlagStore）共享状态条件。

```
┌─────────────────────────────────────────────┐
│                  UI 层                       │
│  HUD / 对话UI / 遭遇UI / 菜单 / 书架 / ... │
├─────────────────────────────────────────────┤
│               系统层（Systems）               │
│  场景 / 对话 / 任务 / 规矩 / 物件 / 遭遇    │
│  / 演出 / 日程 / 档案 / NPC / 音频           │
├─────────────────────────────────────────────┤
│               渲染层（Rendering）             │
│  PixiJS渲染器 / 摄像机 / 精灵实体 / 动画     │
├─────────────────────────────────────────────┤
│               核心层（Core）                  │
│  Game主循环 / 状态机 / 事件总线 / 资源管理    │
│  / 存档管理 / 全局标记(FlagStore) / 输入      │
├─────────────────────────────────────────────┤
│               数据层（Data）                  │
│  JSON数据文件 / ink对话脚本 / 图片 / 音频     │
└─────────────────────────────────────────────┘
```

**架构原则（铁律）：**

1. **分层依赖**：上层依赖下层，下层不依赖上层
2. **系统解耦**：同层系统之间不持有彼此引用，不直接互调。系统间通信只有两条路径：EventBus 发布/订阅事件、FlagStore 读写标记
3. **依赖注入**：每个系统通过构造函数接收所需依赖（EventBus、FlagStore 等），不使用全局单例或跨模块直接 import 其他系统实例
4. **数据驱动**：所有游戏内容由数据文件定义，代码只实现通用逻辑。代码中不出现具体的物件名、规矩名、NPC名、任务ID、对话文本等内容
5. **统一条件源**：FlagStore 是所有条件判断的唯一数据源。所有运行时状态（物件持有、规矩习得、任务进度等）由各系统同步写入 FlagStore，条件判断只查 FlagStore
6. **统一动作执行**：遭遇结果、演出指令、对话触发的游戏行为，统一通过 ActionExecutor 分发，不在各系统中分别硬编码动作处理逻辑
7. **接口约定**：所有系统实现统一的 IGameSystem 接口，保证生命周期和存档行为一致
8. **完整生命周期**：所有系统和实体必须支持完整的创建-运行-销毁周期。destroy() 后不留任何状态残留（渲染对象、事件监听、定时器、缓存引用）。重新 init() 后的行为必须和首次 init() 完全一致。这不仅适用于场景，适用于每一个系统

## 三、核心层（Core）

### 3.0 IGameSystem — 系统接口定义与当前约定

`IGameSystem` 与 `GameContext` 当前定义在 `src/data/types.ts` 中：

```typescript
interface IGameSystem {
  init(ctx: GameContext): void;
  update(dt: number): void;
  serialize(): object;
  deserialize(data: object): void;
  destroy(): void;
}
```

- 当前 `GameContext` 包含 `EventBus`、`FlagStore`、`StringsProvider`
- Game 主类在 `start()` 中按顺序组装系统与 UI，在 `tick()` 中按游戏状态手动调用需要更新的模块
- 系统初始化、更新、存档分发顺序都在 `Game.ts` 中手动编排，不做自动注册框架
- `serialize/deserialize` 由 `SaveManager` 间接调用；所有系统已实现完整序列化
- 当前代码中系统类以“显式声明”为主，全部统一显式 `implements IGameSystem`

### 3.1 Game — 主入口与游戏循环

职责：初始化所有系统、运行游戏主循环（requestAnimationFrame）、管理全局游戏状态机。

游戏状态机：

```
MainMenu ←→ Exploring ←→ Dialogue
                ↕            ↕
            Encounter    Cutscene
                ↕
           UIOverlay（任务面板/背包/书架/地图/暂停等）
```

- **Exploring**：玩家可自由移动和交互，是默认游玩状态
- **Dialogue**：对话进行中，冻结移动，对话UI接管输入
- **Encounter**：遭遇进行中，冻结移动，遭遇UI接管输入
- **Cutscene**：演出播放中，所有玩家输入被拦截（除"点击继续"）
- **UIOverlay**：打开任何全屏UI面板时进入，暂停游戏逻辑，关闭后回到之前状态
- **MainMenu**：主菜单/暂停菜单

每帧（tick）调用顺序：InputManager → 当前状态的更新逻辑 → 各系统更新 → Renderer渲染

### 3.2 EventBus — 事件总线

职责：系统间的解耦通信。所有系统通过发布/订阅事件交互，而非直接引用。

接口：
- `emit(eventName, payload)` — 发布事件
- `on(eventName, callback)` — 订阅事件
- `off(eventName, callback)` — 取消订阅

核心事件清单（按类别）：

**场景相关：**
- `scene:enter` — 进入新场景，payload: { sceneId, fromSceneId }
- `scene:ready` — 场景加载完成

**交互相关：**
- `hotspot:triggered` — 热区触发交互
- `hotspot:pickup:done` — 拾取完成，payload: { hotspotId }
- `hotspot:inspected` — 检查完成，payload: { hotspotId }
- `npc:interact` — NPC 交互触发

**对话相关：**
- `dialogue:start` — 对话开始，payload: { npcName }
- `dialogue:end` — 对话结束
- `dialogue:line` — 对话行输出
- `dialogue:willEnd` — 对话即将结束
- `dialogue:advance` — 推进对话
- `dialogue:advanceEnd` — 推进对话结束
- `dialogue:choices` — 对话选项出现
- `dialogue:choiceSelected` — 玩家做出对话选择，payload: { index }
- `dialogue:choiceSelected:log` — 对话选择记录

**任务相关：**
- `quest:accepted` — 新任务接受，payload: { questId, title }
- `quest:completed` — 任务完成，payload: { questId, title }

**规矩相关：**
- `rule:acquired` — 获得完整规矩，payload: { ruleId, name }
- `rule:fragment` — 获得规矩碎片，payload: { fragmentId, ruleId }
- `rule:used` — 使用规矩，payload: { ruleId }

**物件相关：**
- `item:acquired` — 获得物件，payload: { itemId, count }
- `item:consumed` — 消耗物件，payload: { itemId, count }
- `currency:changed` — 铜钱变化，payload: { amount, newTotal }
- `inventory:discard` — 丢弃物件

**遭遇相关：**
- `encounter:start` — 遭遇开始，payload: { encounterId }
- `encounter:narrative` — 遭遇前置叙事
- `encounter:options` — 遭遇选项出现
- `encounter:result` — 遭遇结果文本
- `encounter:end` — 遭遇结束

**演出相关：**
- `cutscene:start` — 演出开始，payload: { id }
- `cutscene:end` — 演出结束，payload: { id }

**区域相关：**
- `zone:enter` — 进入区域
- `zone:exit` — 离开区域
- `zone:ruleAvailable` — 区域规矩可用
- `zone:ruleUnavailable` — 区域规矩不可用

**时间相关：**
- `day:end` — 一天结束
- `day:start` — 新一天开始，payload: { dayNumber }

**标记相关：**
- `flag:changed` — 标记变化，payload: { key, value }

**档案相关：**
- `archive:updated` — 档案更新，payload: { bookType, entryId }

**规矩使用：**
- `ruleUse:apply` — 请求执行规矩效果，payload: { ruleId, actions, resultText? }
- `ruleUse:showResult` — 显示规矩使用结果

**商店/菜单：**
- `shop:purchase` — 商店购买
- `shop:closed` — 商店关闭
- `menu:newGame` — 新游戏
- `menu:returnToMain` — 返回主菜单

**通用：**
- `notification:show` — 显示角落通知，payload: { text, type }
- `map:travel` — 地图传送请求

### 3.3 FlagStore — 全局标记系统

职责：存储所有游戏状态标记，是所有条件判断的唯一数据源。

接口：
- `set(key, value)` — 设置标记（布尔或数字），自动触发 `flag:changed` 事件
- `get(key)` — 读取标记值
- `checkConditions(conditions)` — 检查条件数组，全部满足返回 true
- `serialize()` / `deserialize(data)` — 存档/读档

条件数组格式（隐式 AND，所有条件必须同时满足）：

```json
[
  { "flag": "quest_01_done" },
  { "flag": "coins", "op": ">=", "value": 5 },
  { "flag": "rule_03_acquired", "value": true }
]
```

- 省略 `op` 时默认为 `==`
- 省略 `value` 时默认为 `true`（适用于布尔标记的快速写法）

所有系统在需要判断条件时调用 `flagStore.checkConditions()`，包括：对话分支、遭遇选项显示、NPC状态、热区可用性、任务触发等。

FlagStore的key命名规范：
- 任务相关：`quest_{id}_status`（值：0未触发/1进行中/2完成）
- 规矩相关：`rule_{id}_acquired`（布尔）
- 物件相关：由InventoryManager在增减时同步写入，如 `has_item_{id}`
- 剧情相关：`story_{name}`（布尔或数字）
- 日期相关：`current_day`（数字）

### 3.4 AssetManager — 资源管理

职责：加载、缓存、提供所有游戏资源的统一访问。

- 图片：通过PixiJS的Assets API加载纹理和图集
- 音频：通过Howler.js预加载音频文件
- 数据：加载JSON数据文件并解析为TypeScript对象
- ink脚本：加载编译后的ink JSON

提供按场景预加载的能力：在场景切换前，预加载目标场景需要的所有资源。

### 3.5 SaveManager — 存档管理

职责：收集所有系统状态、序列化为JSON、存储到localStorage（或文件系统）。

存档流程：
1. 向各系统收集可序列化状态（每个系统实现 `serialize()` 方法）
2. 合并为一个完整的存档对象
3. JSON序列化后存入对应槽位

读档流程：
1. 从槽位读取JSON并反序列化
2. 将各部分分发给对应系统（每个系统实现 `deserialize(data)` 方法）
3. 重新加载当前场景

### 3.6 GameStateController

职责：集中管理游戏状态机、UI 面板注册与切换、键盘快捷键绑定、Escape 键处理。

从 `Game.ts` 提取而出，避免主类成为 God Object。核心能力：

- `setState(newState)` / `restorePreviousState()` — 游戏状态切换
- `registerPanel(name, panel, shortcutKey?)` — 注册 UI 面板及对应快捷键
- `togglePanel(name)` — 切换面板开关状态，自动管理 Exploring ↔ UIOverlay 状态转换
- `setEscapeFallback(fn)` — 设置 Escape 键在 Exploring 状态下的回退行为（如打开菜单）
- `destroy()` — 清理 window 键盘监听

面板需实现 `ToggleablePanel` 接口（`isOpen`、`open()`、`close()`）。

### 3.7 InputManager — 输入管理

职责：统一管理键盘输入，并提供按帧查询接口。

- 当前实现提供按键状态记录、`wasKeyJustPressed()` 与 `endFrame()` 等基础能力
- 不同游戏状态的输入分发由 `Game.ts`、`DialogueUI`、`EncounterUI` 等模块手工处理
- 快捷键映射由 `GameStateController` 管理，通过 `registerPanel` 注册面板及对应快捷键

### 3.8 StringsProvider

职责：加载并提供面向玩家的通知文案模板。所有系统层的玩家可见文本（如"新任务：{title}"、"包袱满了！"）从 `strings.json` 读取，支持变量插值，为后续本地化提供基础。

- `load()` — 异步加载 `strings.json`
- `get(category, key, vars?)` — 获取文案模板，支持 `{varName}` 变量替换

### 3.9 ActionExecutor — 统一动作执行

职责：提供一个中心化的动作执行函数，将 JSON 数据中描述的"动作"映射到实际的游戏逻辑调用。

实现方式：一个 `execute(action)` / `executeBatch(actions)` 接口 + 内部 handler 映射表。

JSON 数据中的动作格式：

```json
{ "type": "giveItem", "params": { "id": "nuomi", "count": 3 } }
{ "type": "setFlag", "params": { "key": "met_blind_li", "value": true } }
{ "type": "removeCurrency", "params": { "amount": 5 } }
{ "type": "giveRule", "params": { "id": "rule_03" } }
{ "type": "showNotification", "params": { "text": "规矩本新增", "type": "rule" } }
{ "type": "startCutscene", "params": { "id": "prologue_opening" } }
```

已知动作类型（初始集合，按需扩充）：

| 动作类型 | 参数 | 说明 |
|---------|------|------|
| giveItem | id, count | 给予物件 |
| removeItem | id, count | 扣除物件 |
| giveCurrency | amount | 给予铜钱 |
| removeCurrency | amount | 扣除铜钱 |
| giveRule | id | 获得完整规矩 |
| giveFragment | id | 获得规矩碎片 |
| setFlag | key, value | 设置全局标记 |
| updateQuest | id | 更新任务状态 |
| addArchiveEntry | bookType, entryId | 添加档案条目 |
| startEncounter | id | 触发遭遇 |
| playBgm | id, fadeMs | 播放背景音乐 |
| stopBgm | fadeMs | 停止背景音乐 |
| playSfx | id | 播放音效 |
| showNotification | text, type | 显示角落通知 |
| endDay | - | 结束当前天 |
| addDelayedEvent | targetDay, actions | 添加延迟事件 |
| startCutscene | id | 播放演出 |
| openShop | shopId | 打开商店 |

统一使用场景：遭遇结果、区域进出动作、对话标签动作、任务奖励发放、延迟事件执行、演出中的 `execute_action`。主入口 `Game.ts` 会在基础 handler 之上补充注册游戏相关动作。

## 四、渲染层（Rendering）

### 4.1 Renderer

职责：封装PixiJS，管理舞台（Stage）和各渲染层级。

渲染层级（从下到上）：
1. **背景层**（Background）：多张背景图叠加
2. **实体层**（Entity）：角色、NPC、场景物体。按Y轴排序
3. **前景层**（Foreground）：遮挡角色的前景元素（树枝、屋檐等）
4. **UI层**（UI）：HUD、对话框、选项面板等

### 4.2 CutsceneRenderer

职责：封装演出系统所需的所有渲染操作，使 CutsceneManager（系统层）无需直接操作 PixiJS 对象。

提供方法：
- 淡入淡出（fadeToBlack/fadeFromBlack/flashWhite）
- 标题卡显示（showTitle）
- 演出对话框显示/关闭（showDialogueBox/dismissDialogueBox）
- 表情气泡（showEmoteBubble）
- 摄像机移动/缩放动画（cameraMove/cameraZoom）
- 实体添加到场景（addToEntityLayer）
- 清理（cleanup）

### 4.3 Camera

职责：控制视口位置和缩放。

- 跟随目标（默认跟随玩家角色）
- 平滑移动（lerp插值跟随）
- 场景边界限制（不超出场景范围）
- 供演出系统调用：移动到指定位置、缩放

### 4.4 SpriteEntity

职责：带动画能力的精灵实体基类，供Player、NPC等继承。

- 持有一个AnimatedSprite
- 提供播放指定动画状态的接口
- 自动管理朝向翻转
- 支持沿路点序列移动（供演出和NPC路径使用）

## 五、系统层（Systems）

### 5.1 SceneManager

- 维护场景数据缓存和当前场景引用
- 加载场景：解析JSON → 创建占位背景 → 创建热区 → 放置NPC → 配置音频/区域 → 放置玩家角色
- 场景切换：卸载当前场景 → 过渡效果 → 加载新场景
- 场景状态记忆：当前记录已检查/已拾取热区，切场景时保存并在再次加载时恢复
- 通过回调把碰撞、玩家出生点、摄像机边界、场景音频、区域数据、交互对象（热区/NPC）交给其他模块

### 5.2 InteractionSystem

- 每帧检测玩家位置与所有活跃热区的距离
- 进入交互范围时通知UI层显示交互提示
- 收到交互输入时，根据热区类型分发：触发对话 / 弹出检查文字 / 拾取物品 / 切换场景
- 检查热区的前置条件（通过FlagStore）

### 5.3 DialogueManager

- 持有inkjs的Story实例
- 开始对话：加载指定对话脚本节点，切换游戏状态到Dialogue
- 推进对话：读取下一行文字，传给DialogueUI显示
- 处理选择：读取ink的choices，传给DialogueUI显示选项，接收玩家选择
- 动作触发：解析 ink 标签中的 `action:*` 指令（如 `giveItem`、`setFlag`、`giveRule` 等），并转发给 ActionExecutor 执行
- `getFlag(key)` 函数供 ink 脚本查询 FlagStore 标记值
- `dialogue:start` 事件 payload 为 `{ npcName }`（NPC 名称字符串），用于档案等模块做解锁判断

### 5.4 QuestManager

- 维护所有任务的状态列表
- 主线任务链：完成当前 → 自动激活下一个
- 当前主要监听 `flag:changed`，每次重新评估活跃任务完成条件与未触发任务前置条件
- 完成时发放奖励并触发 `quest:completed` 事件
- 支线任务触发：在评估时检查未触发任务的前置条件

### 5.5 RulesManager

- 维护已学规矩列表和碎片列表
- 获取规矩：完整获取 / 碎片合成
- 碎片管理：每次获得碎片后检查是否可合成
- 提供规矩定义查询、是否已习得、是否已发现、碎片进度等接口
- 同步到FlagStore：每次规矩变化时更新 `rule_{id}_acquired` 标记

### 5.6 InventoryManager

- 维护背包物件列表和铜钱数量
- 增/减/查物件，容量检查
- 铜钱增/减/查
- 同步到FlagStore：物件变化时更新 `has_item_{id}` 标记，铜钱变化时更新 `coins` 标记
- 关键道具的动态描述：根据FlagStore标记返回对应描述版本

### 5.7 EncounterManager

- 加载遭遇数据
- 触发遭遇：切换游戏状态 → 显示前置叙事 → 生成选项
- 选项生成逻辑：
  1. 加入所有通用选项
  2. 遍历遭遇数据中的规矩选项和特殊选项，通过 `FlagStore.checkConditions()` 检查每个选项的前置条件（规矩是否习得、道具是否持有等），满足条件的选项才显示
  3. 对需要消耗道具的选项，通过 FlagStore 检查道具数量，不足则标记为灰色不可用
- 执行选择结果：将结果中的动作列表交给 ActionExecutor 批量执行

### 5.8 DayManager

- 维护当前天数
- 天结束流程：触发 `day:end` → 处理延迟事件队列 → 递增天数 → 触发 `day:start`
- 延迟事件队列：添加事件（天数+内容）、到期检查、执行到期事件

### 5.9 CutsceneManager

- 加载演出数据（时间线指令列表）
- 播放演出：切换游戏状态到Cutscene → 按时间线顺序执行指令
- 指令执行器：每种指令类型有一个执行函数
- 支持 `parallel` 标记的并行执行
- 当前内置指令包含：淡入淡出、等待、标题、对话框、BGM/SFX、摄像机移动/缩放、临时实体生成/移动/朝向/表情/显隐、切场景事件、`execute_action`
- 所有渲染操作（淡入淡出、标题卡、对话框、表情气泡、摄像机动画）委托给渲染层的 `CutsceneRenderer`，CutsceneManager 本身不直接创建 PixiJS 对象
- 演出结束后恢复游戏状态

### 5.10 ArchiveManager

- 维护人物簿、见闻录、杂书匣、独立书籍的数据与解锁状态（规矩本仍由 `RulesManager` 管理）
- 人物簿：监听 `dialogue:start`，按 NPC 标识尝试解锁
- 见闻录 / 杂书匣：主要在 `flag:changed` 后按条件计算解锁
- 独立书籍：支持整本解锁与按页条件显示
- 所有更新触发 `archive:updated` 事件

### 5.11 ZoneSystem

- 维护当前场景中的区域列表与玩家所在区域集合
- 玩家进入/离开区域时执行 `onEnter` / `onExit` 动作列表
- 通过 `zone:ruleAvailable` / `zone:ruleUnavailable` 通知 UI 是否可使用规矩
- 提供当前激活区域中的 `ruleSlots` 给 `RuleUseUI`

### 5.12 AudioManager

- BGM控制：播放、停止、淡入淡出切换
- 环境音控制：多层叠加、独立音量
- 音效播放：一次性触发
- 供各系统和演出系统调用

## 七、数据流关键路径

### 7.1 对话 → 获取规矩 → 遭遇中使用

```
玩家与NPC对话
  → DialogueManager 运行 ink 脚本
  → 读取到标签 # action:giveRule:rule_03
  → DialogueManager 转发给 ActionExecutor.executeAction({ type: "giveRule", params: { id: "rule_03" } })
  → ActionExecutor 调用 giveRule handler → RulesManager 更新规矩本，写入 FlagStore，emit "rule:acquired"
  → NotificationUI 收到事件，显示"规矩本新增"

后续：玩家进入遭遇
  → EncounterManager 加载遭遇数据
  → 遍历遭遇选项，通过 FlagStore.checkConditions() 检查每个选项的条件
  → rule_03_acquired == true → 对应的规矩选项显示
  → EncounterUI 显示选项面板（含规矩选项）
```

### 7.2 遭遇选择 → 延迟后果 → 次日生效

```
玩家在遭遇中做出选择
  → EncounterManager 将结果动作列表交给 ActionExecutor 批量执行
  → 其中一个动作：{ type: "addDelayedEvent", params: { targetDay: currentDay + 1, actions: [{ type: "setFlag", params: { key: "nightmare_01", value: true } }] } }
  → ActionExecutor 调用 addDelayedEvent handler → DayManager 将事件加入队列

当天主线推进触发天结束
  → DayManager 触发 day:end
  → 处理延迟队列：找到 day == nextDay 的事件
  → 将到期事件的 actions 交给 ActionExecutor 执行 → FlagStore.set("nightmare_01", true)
  → 后续系统/UI 继续通过 FlagStore.checkConditions(...) 读取新的状态
```

### 7.3 任务完成流程

```
玩家与NPC对话完成
  → DialogueManager emit "dialogue:end"
  → QuestManager 收到事件，检查活跃任务
  → 主线任务 "拜师学艺" 的完成条件 = [{ "flag": "dialogue_blind_li_01_done" }]
  → FlagStore.checkConditions(...) → true
  → QuestManager 完成任务：发铜钱+物件，emit "quest:completed"
  → 自动激活下一个主线任务 "七日之约"
  → NotificationUI 显示"任务完成"和"新任务"
```

## 九、配置驱动清单

明确划分哪些内容由配置/数据文件驱动，哪些允许硬编码在程序中。

**必须由配置驱动（代码中禁止硬编码）：**

- 场景的一切内容：背景图层、碰撞区域、交互热区、NPC放置、音频配置、遭遇触发区域

**场景规范：**

- 场景边界默认与场景背景图同尺寸：若场景 JSON 中未写 `width`/`height`，加载时以首张背景图尺寸作为场景宽高。
- 可为场景设定 `backgroundScale`（默认 1）：场景逻辑尺寸 = 首张背景图尺寸 × `backgroundScale`；背景图绘制时也按该系数缩放。未写 `width`/`height` 时由首张背景图尺寸 × `backgroundScale` 推导；写了 `width`/`height` 时以 JSON 为准，仅绘制时对背景应用 `backgroundScale`。
- 场景内 NPC 的基础属性：名称、位置、对话文件、交互范围
- 所有对话文本和分支逻辑（ink 脚本）
- 所有任务的定义：标题、描述、前置条件、完成条件、奖励、链式关系
- 所有规矩和碎片的定义：名称、描述、分类、来源、关联遭遇/对话
- 所有物件的属性：名称、描述、价格、最大叠加数
- 所有遭遇的定义：触发条件、选项列表、选项条件、结果动作
- 所有演出的时间线：指令类型、参数、时序、依赖关系
- 所有档案数据：人物簿条目、见闻录条目、文档内容、独立书籍页面
- 玩家动画配置：帧尺寸、状态帧范围、帧率、是否循环
- 音频映射：场景-BGM/环境音映射、音效ID映射
- 商店售卖清单与价格
- 地图节点名称、位置、解锁条件

**允许硬编码（结构性的，数量固定，不随内容增减而变化）：**

- 游戏状态机的状态枚举和转换规则
- 渲染层级定义（背景层/实体层/前景层/UI层）
- ActionExecutor 的 handler 映射表（动作类型是有限集合）
- checkConditions 的比较运算符支持集合
- UI 面板的注册列表、布局结构与快捷键绑定
- 系统初始化和更新顺序

## 十、扩展性示例

以下示例说明如何纯通过数据扩展游戏内容，不改动已有代码：

**添加一个新 NPC：**
1. 在 `npcs.json` 中添加 NPC 数据条目（ID、名称、位置、外观、状态规则）
2. 编写新的 `.ink` 对话脚本
3. 在对应场景 JSON 中添加 NPC 放置点
4. 放入精灵图资源 → 零代码改动

**添加一条新规矩：**
1. 在 `rules.json` 中添加规矩定义（ID、名称、分类、描述、关联遭遇ID列表）
2. 在相关遭遇 JSON 中添加使用该规矩的选项
3. 在相关 ink 对话中添加 `giveRule("new_rule_id")` 调用 → 零代码改动

**添加一个新遭遇：**
1. 在 `encounters.json` 中定义遭遇（选项、条件、结果动作）
2. 结果动作使用已有的 ActionExecutor 动作类型组合
3. 在对应场景 JSON 中添加遭遇触发区域 → 零代码改动

**添加一种新的动作类型（极少发生）：**
1. 在 ActionExecutor 的 handlers 映射表中添加一条：类型名 → 处理函数
2. 在 JSON 数据中使用新类型 → 仅改动一个文件的一处
