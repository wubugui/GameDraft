## 0. 验收前准备

- [ ] 当前仓库已经拉到最新 `master`。
- [ ] 工作区没有未提交改动，或已经确认这些改动不影响验收。
- [ ] 本机存在 `.tools\Python311\python.exe`。
- [ ] 已安装 npm 依赖；如果不确定，先运行一次 `npm install`。
- [ ] 验收过程中记录所有失败项，优先复制工具内报告，不只截图。

建议先运行：

```powershell
git status --short --branch
npm install
```

通过标准：

- `git status --short --branch` 不显示异常冲突。
- `npm install` 能正常完成，或依赖已经是最新。

## 1. 启动入口验收

### 1.1 通过脚本启动

步骤：

1. 在资源管理器打开项目根目录。
2. 双击 `start-production-workbench.cmd`。
3. 等待窗口出现。
4. 检查窗口标题是否类似 `GameDraft 生产工作台 - GameDraft`。
5. 检查顶部工程路径是否指向当前项目。
6. 关闭窗口。

通过标准：

- [ ] 窗口能正常打开。
- [ ] 没有 Python import 错误。
- [ ] 工程路径正确。
- [ ] 关闭窗口没有报错弹窗。

失败时记录：

- 复制或截图错误弹窗。
- 记录是否缺 Python、PySide6、路径错误或窗口无响应。

### 1.2 通过 npm 命令启动

步骤：

1. 在项目根目录打开 PowerShell。
2. 运行：

```powershell
npm run planner:gui
```

3. 等待生产工作台窗口出现。
4. 检查页签是否包含：
   - `剧情单元`
   - `每日检查`
   - `Graph诊断`
   - `运行时Debug`
   - `素材审计`
   - `素材候选`
   - `图片工具`
   - `动画Sheet`
   - `素材任务`
   - `Codex/GPT`

通过标准：

- [ ] 命令能启动 GUI。
- [ ] 页签完整。
- [ ] 没有空白窗口或卡死。

### 1.3 通过主编辑器入口启动

步骤：

1. 启动原有 Python 编辑器。
2. 打开菜单 `Tools -> External tools (new process) -> Production Workbench`。
3. 等待生产工作台窗口出现。
4. 确认主编辑器仍可操作。
5. 关闭生产工作台，确认主编辑器没有被一起关闭。

通过标准：

- [ ] 菜单入口存在。
- [ ] 点击后能打开生产工作台。
- [ ] 生产工作台是新进程，不会卡住主编辑器。
- [ ] 关闭生产工作台不影响主编辑器。

## 2. 每日检查验收

步骤：

1. 打开生产工作台。
2. 进入 `每日检查` 页签。
3. 点击 `运行每日检查`。
4. 检查按钮是否变成 `检查运行中...`。
5. 等待检查完成。
6. 查看输出结果。
7. 点击 `复制报告`。
8. 粘贴到临时文本里，确认剪贴板里是完整报告。
9. 点击顶部 `打开报告目录`，确认报告目录能打开。

通过标准：

- [ ] 检查过程不冻结整个窗口。
- [ ] 检查完成后按钮恢复为 `运行每日检查`。
- [ ] 报告中包含 `每日检查` 总结。
- [ ] 报告中包含以下通过项：
  - [ ] `Python 编辑器/Narrative smoke`
  - [ ] `生产工作台 smoke`
  - [ ] `Python import smoke`
  - [ ] `TS Narrative/runtime save smoke tests`
- [ ] `error=0`。
- [ ] `blocker=0`。
- [ ] 如有 warning，能看懂 warning 指向的内容或素材问题。
- [ ] `复制报告` 可用。
- [ ] `打开报告目录` 可用，能看到新生成或已有的报告文件。

不通过标准：

- 出现 `error` 或 `blocker`。
- 窗口卡死。
- 报告只有原始 JSON，没有可读总结。
- 复制报告为空。

## 3. 剧情单元验收

### 3.1 列表和摘要

步骤：

1. 进入 `剧情单元` 页签。
2. 点击 `刷新列表`。
3. 在左侧或列表区域选择一个已有剧情单元。
4. 查看右侧摘要、基础信息和状态字段。
5. 切换另一个剧情单元，再确认内容随选择变化。

通过标准：

- [ ] 能加载剧情单元列表。
- [ ] 选择单元后能显示名称、类型、制作状态、入口、出口、验收信息。
- [ ] 切换单元不会残留上一个单元的错误内容。
- [ ] 加载期间关键按钮不会导致误保存旧数据。

### 3.2 操作向导和自检

步骤：

1. 选择一个剧情单元。
2. 点击 `操作向导`。
3. 查看工具给出的下一步建议。
4. 点击 `当前单元自检`。
5. 查看自检报告。
6. 点击 `复制当前单元报告`。
7. 粘贴到临时文本，确认报告包含单元 ID、入口、出口、验收路线、阻塞或下一步建议。

通过标准：

- [ ] `操作向导` 能根据当前状态给出具体下一步。
- [ ] `当前单元自检` 能指出缺入口、缺出口、缺验收路线等问题。
- [ ] `复制当前单元报告` 内容完整，可直接发给 Codex 修问题。

### 3.3 验收路线编辑

步骤：

1. 选择一个还没有完整验收路线的剧情单元。
2. 点击 `生成验收草稿`。
3. 检查是否生成验收路线。
4. 如果无法生成，确认提示说明了缺少 dialogue / signal / state 等可推断信息。
5. 使用选择器添加验收起点、步骤或期望结果。
6. 点击 `复制验收路线`。
7. 检查复制内容是否可读。
8. 点击 `1. 检查脚本`。

通过标准：

- [ ] 能生成草稿，或给出明确不能生成的原因。
- [ ] 路线步骤能通过选择器添加，不要求手写复杂 ID。
- [ ] `复制验收路线` 可用。
- [ ] `1. 检查脚本` 能给出通过或具体错误。

### 3.4 保存规则

步骤：

1. 修改一个剧情单元的备注或素材需求。
2. 点击 `保存当前单元`。
3. 重新点击 `刷新列表`。
4. 再次选择该单元，确认内容仍在。
5. 尝试把制作状态改为 `待验收`，但故意留空入口、出口或验收路线。
6. 点击保存。

通过标准：

- [ ] 正常字段修改能保存。
- [ ] 刷新后内容不丢失。
- [ ] 状态切到 `待验收`、`通过`、`冻结` 时会严格校验必填项。
- [ ] 缺入口、出口、验收路线或最近验收未通过时，不允许错误进入收口状态。

## 4. Graph 诊断验收

步骤：

1. 进入 `Graph诊断` 页签。
2. 点击 `刷新 Graph 诊断`。
3. 等待诊断完成。
4. 查看输出是否包含 signal flow、flag read/write、quest dependency 或 runtime trace 相关信息。
5. 点击 `选择范围`，选择一个剧情单元范围。
6. 再点击 `全部`，恢复全量诊断。
7. 点击 `打开相关源`，选择一个范围后确认能打开或定位相关源文件。
8. 点击 `复制报告`，粘贴检查内容。

通过标准：

- [ ] 诊断能完成。
- [ ] 范围筛选可用。
- [ ] `全部` 能恢复全量视图。
- [ ] `打开相关源` 能找到对应 narrative graph、composition 或相关源。
- [ ] 复制报告不是空白。
- [ ] 报告以可读诊断呈现，不只是大段原始 JSON。

不通过标准：

- 刷新后崩溃。
- 范围筛选后内容错乱。
- 找不到明显存在的剧情单元源文件。

## 5. 运行时 Debug 验收

验收前准备：

1. 启动游戏运行时：

```powershell
npm run dev
```

2. 在浏览器打开 Vite 提供的本地地址。
3. 让游戏至少进入一个可交互场景。
4. 保持游戏运行，再打开生产工作台。

### 5.1 快照和 trace

步骤：

1. 进入 `运行时Debug` 页签。
2. 点击 `刷新运行时快照`。
3. 查看是否显示最新 Debug 快照和 Runtime Trace。
4. 点击 `请求 runtime 快照`。
5. 再次点击 `刷新运行时快照`。
6. 点击 `清 runtime trace`。
7. 再次刷新，确认 trace 被清理或重新开始。
8. 点击 `复制事故报告`，粘贴检查。

通过标准：

- [ ] 能看到运行时快照。
- [ ] 能请求新的 runtime 快照。
- [ ] 能清理 runtime trace。
- [ ] 事故报告包含当前场景、状态、trace 或错误上下文。
- [ ] 游戏页面不因请求快照崩溃。

### 5.2 命令队列

步骤：

1. 在 `运行时Debug` 页签点击 `查看命令队列`。
2. 检查队列内容是否可读。
3. 点击 `清空命令队列`。
4. 再次查看队列。

通过标准：

- [ ] 能查看命令队列。
- [ ] 能清空命令队列。
- [ ] 清空后不会影响游戏正常运行。

### 5.3 剧情单元发送到游戏运行

步骤：

1. 保持游戏运行。
2. 回到 `剧情单元` 页签。
3. 选择一个有验收路线的剧情单元。
4. 点击 `1. 检查脚本`。
5. 检查通过后点击 `2. 发送到游戏运行`。
6. 回到浏览器观察游戏是否执行对应操作。
7. 点击 `只对比当前快照`。
8. 点击 `3. 完成并记录结果`。
9. 查看最近验收结果是否更新为 `通过`、`失败` 或 `阻塞`。

通过标准：

- [ ] 脚本检查能发现明显错误。
- [ ] 发送到游戏后，运行时命令进入队列并被游戏处理。
- [ ] 当前快照对比能返回可读结果。
- [ ] 完成验收后会保存最近验收结果和备注。
- [ ] 失败时报告足够复现问题。
