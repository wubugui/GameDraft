1. 系统定义

层次叙事系统是一套多层叙事状态机运行时。它把 Flow、NPC、Hotspot、Zone、Quest、Scenario 等叙事对象组织为状态子图。每个子图记录当前叙事状态，Transition 描述状态迁移关系，Signal 推动迁移发生，生命周期 Action 把状态变化投影到现有运行时系统。

叙事状态机负责管理叙事状态、状态迁移、状态生命周期 Action、跨子图因果、状态镜像和存档；编辑器负责把触发、读取、效果等关系投影成可视化边。

2. 总体架构与运行流程

状态机运行在现有系统上方。现有系统发出 signal 或 setState；状态机根据当前 activeState、signal 和 conditions 匹配 Transition；状态生命周期 Action 继续复用现有 ActionExecutor；旧系统通过状态镜像或查询 API 读取叙事状态。

emitNarrativeSignal 流程

1外部来源

Dialogue、Zone、Minigame、Action 发出 signal。

2进入队列

所有 signal / trigger / setState 串行排队。

3匹配迁移

检查 activeState、Transition.signal、conditions、priority。

4执行生命周期

old.onExit → activeState = to → new.onEnter。

5镜像与传播

更新 narrative.*.active，并派发 stateEntered。

运行时规则

队列规则

所有 signal / trigger / setState 请求进入同一个队列。队列串行处理，避免多个来源交错改写状态。

迁移规则

一次 signal / trigger 处理中，每个 Graph 最多迁移一次。候选 Transition 按 priority 排序，priority 相同按声明顺序。

级联规则

状态迁移产生 stateEntered / stateExited trigger，可继续推动其他 Graph。级联传播设置最大步数。

Action 失败

生命周期 Action 失败记录为运行时错误，状态迁移结果保持成立，队列继续处理。

3. 核心数据模型

底层实现保持简单：Graph 持有 State 和 Transition；Transition 自己携带 signal、条件和优先级；Signal 可以由 DialogueGraph、Zone、Minigame、Action、stateEntered 等来源产生。

NarrativeGraph

type NarrativeGraph = { id: string; ownerType: 'flow' | 'npc' | 'hotspot' | 'zone' | 'quest' | 'scenario' | 'system'; ownerId?: string; initialState: string; states: Record<string, NarrativeStateNode>; transitions: NarrativeTransition[]; };

NarrativeStateNode

type NarrativeStateNode = { id: string; label?: string; description?: string; onEnterActions?: ActionDef[]; onExitActions?: ActionDef[]; meta?: Record<string, unknown>; };

NarrativeTransition

type NarrativeTransition = { id: string; from: string; to: string; signal?: string; conditions?: ConditionExpr[]; priority?: number; };

NarrativeSignal

type NarrativeSignal = { sourceType: 'dialogue' | 'zone' | 'minigame' | 'cutscene' | 'quest' | 'action' | 'entity' | 'state' | 'system'; sourceId: string; signal: string; };

Transition 匹配规则

顺序规则
1只检查当前 Graph 的 activeState 等于 from 的 Transition。
2输入 signal 与 Transition.signal 匹配，且 conditions 满足时，Transition 成为候选。
3候选按 priority 降序排列；未配置 priority 时按 0 处理。
4priority 相同则按 transitions 数组中的声明顺序排列。
5每个 Graph 对一次 signal 只执行第一条候选 Transition。

状态镜像

NarrativeState 与普通 Flag 同时存在。Narrative state 是叙事状态机子系统自己的序列化数据；mirror flag 是对外读取入口。

narrative.<graphId>.<stateId>.active narrative.npc_ringboy.after_event.active = true narrative.flow_dock_water_monkey.crate_minigame_done.active = true

5. 实体与子系统接入

有状态实体：暴露实体状态

复杂 NPC、特殊 Hotspot、AI 或有内部状态机的实体，可以直接把自己的状态暴露给叙事层。实体逻辑也可以根据自身结果 emit signal 或 setState。

选择有状态实体 声明该实体暴露哪些状态 映射到 NarrativeGraph / State 在 Transition / Condition 中引用 实体逻辑 emit signal 或 setState

无状态实体：创建叙事包裹对象

普通场景 NPC、Hotspot、Zone 可以由 Narrative Wrapper Graph 包裹。包裹对象持有 activeState，并通过 onEnter/onExit Action 操作被绑定的实体。

选择无状态实体 创建 Narrative Wrapper Graph 绑定全局唯一实体 id 定义 State 和 Transition 配置 onEnterActions / onExitActions

Quest / Scenario / DialogueGraph

子系统接入方式设计要点
Scenario直接接入叙事层Scenario 本身接近轻量子图。line 可映射为 Graph，phase 可映射为 State 或输入状态。
QuestQuest Wrapper GraphQuest 当前更像 Action 驱动 + 自条件驱动系统。用 wrapper 管理它在大叙事中的位置。
DialogueGraph可执行黑盒内部 currentNode / waitingChoice 归 DialogueGraphManager；叙事层只接收明确 narrative signal，并允许 DialogueGraph 读取叙事状态做分支。

8. 编辑器如何操作、如何响应

作者操作编辑器响应运行时含义
在大画布添加 Wrapper Graph生成一个可展开的叙事子图节点，并要求绑定全局唯一实体 id。NPC、Hotspot、Quest 等通过 wrapper activeState 接入叙事层。
在大画布添加 Scenario显示为 Scenario 子图节点，可展开查看 line / phase 映射状态。Scenario 状态直接暴露到 NarrativeStateManager。
在大画布添加 DialogueGraph显示为黑盒节点，右侧显示它读取哪些状态、会发出哪些 narrative signal。DialogueGraph 内部执行状态归 DialogueGraphManager；叙事层只接收 signal。
连接两个主 Graph 状态生成 Transition，右侧编辑 from、to、signal、conditions、priority。运行时根据 activeState、signal、conditions 匹配迁移。
开启推导触发图层根据 Action、DialogueBridge signal、Transition.signal、stateEntered 等配置显示触发来源。这是自动分析结果，作者不手工连接这类边。
开启推导读取图层根据 getActiveState / isStateActive / 条件配置 / 脚本读取点显示读取关系。这是自动分析结果，作者不手工连接这类边。
点击“导航到绑定实体”右侧显示目标实体 id、场景、类型，并给出跳转入口。叙事编排在本编辑器完成；实体细节可以导航到关联编辑器。
模拟 emit signal更新 activeState、日志和高亮节点；可能级联触发其他 Graph。等价于运行时队列处理一次 signal。
