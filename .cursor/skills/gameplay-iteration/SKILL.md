# Gameplay Iteration

用于处理会影响玩法设计、玩家体验或内容定义的迭代。先审查玩法一致性，再改文档，再改代码，最后做局部回归 review。

## 何时使用

当需求涉及以下任一内容时使用本技能：

- 任务、支线、主线推进
- 遭遇、规矩、规矩碎片、规矩使用
- 对话分支、对话奖励、对话条件
- 物件、货币、购买、掉落、奖励
- 场景交互、热区、场景解锁、探索反馈
- 玩家可见的 UI 交互规则或结果逻辑
- 任何会改变玩家结果、资源流、进度流、玩法节奏的改动

如果任务最终会改变 `玩法功能需求清单.md` 的含义或实现路径，也使用本技能。

## 必须遵循的流程

### 1. 先判定是否属于玩法迭代

- 先快速判断需求是否影响玩法规则、进度结构、数值逻辑、资源消耗、条件判断、对话结果或玩家可见行为。
- 若不影响玩法，只是纯技术实现、重构、工具、性能、架构修复，停止使用本技能，改用 `feature-iteration`。

### 2. 先读玩法文档，再判断冲突

- 先读取 `玩法功能需求清单.md` 的相关章节。
- 再读取 `游戏架构设计文档.md` 的相关章节。
- 必要时再读取已有 review 文档，理解近期约束和已知技术债。

重点判断：

- 新需求是否与现有玩法需求冲突
- 新需求是否让已有玩法定义前后矛盾
- 新需求是否缺少必要决策，无法直接实现

### 3. 有冲突时禁止直接实现

如果发现玩法冲突、需求歧义或存在多个合理方案：

- 不要直接改代码
- 不要擅自替用户做设计取舍
- 输出冲突点、影响范围、可选方案
- 明确等待用户决策

只有在玩法不冲突、方向明确时，才进入下一步。

### 4. 先更新玩法文档，再动代码

如果玩法方向明确且允许实施：

- 先更新 `玩法功能需求清单.md`
- 仅修改与本次迭代直接相关的章节
- 保持原有文档结构和术语一致
- 不顺手重写整个文档

更新完成后，再开始代码实施。

### 5. 读取相关代码并理解实现路径

只读取本次需求直接相关的代码：

- `src/core/`
- `src/systems/`
- `src/ui/`
- `src/rendering/`
- `src/data/`
- `public/assets/data/`
- `public/assets/dialogues/`

避免无关扩散。优先基于已有模式做最小改动。

### 6. 实施时必须遵守架构铁律

始终对照 `游戏架构设计文档.md`，尤其检查：

- UI 不直接执行业务动作
- 同层 system 不直接互相持有引用
- 下层不反向依赖上层
- 玩家内容优先走数据文件，不写死在代码里
- 条件判断统一走 FlagStore
- 游戏行为统一走 ActionExecutor
- 所有系统保持完整生命周期和存档约定

如果实现需求会破坏这些原则，先停下来报告，不要硬做。

### 7. 完成后做局部回归 review

不要做整项目 review，只检查与本次迭代直接相关的范围：

- 改动文件本身是否正确
- 受影响的事件链、动作链、存档链是否闭环
- 是否引入新的跨层依赖、硬编码或职责混杂
- 若改到 UI，确认界面尺寸合适且所有内容都能被看到
- 若改到玩法数据，确认文档、数据、代码三者一致

### 8. 发现问题先自修复，再结束

- 局部 review 发现问题时，先修复问题
- 修复后再次做一次局部复查
- 只有在本次范围内没有明显遗留问题时，才可结束

### 9. 结束时的汇报格式

最终汇报应包含：

- 本次玩法迭代改了什么
- 是否更新了玩法文档
- 改动影响了哪些模块
- 做了哪些局部回归检查
- 是否发现并修复了问题
- 仍然存在的残余风险或待确认点

## 额外限制

- 不要创建任何单独测试用例
- 不要把一次需求扩展成大范围重构
- 不要在冲突未解决时继续实现
- 不要忽略文档更新
- 不要把局部回归 review 偷换成“只要能编译就算完成”
