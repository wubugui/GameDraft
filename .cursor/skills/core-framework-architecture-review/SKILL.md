# 核心框架架构审查

对项目核心框架进行架构审查，对照项目中的架构设计文档所写的架构原则，发现并报告问题。**不主动修复**，仅在用户明确说「开始修复」后按计划逐项修复。

## 审查流程（必须按顺序执行）

### 第一步：探索并理解整个项目代码

- **通过探索项目自行发现所有结构**：不依赖任何预设路径或预设名称。从项目根开始探索，自行发现并归纳：有哪些层、哪些系统、主入口如何组装、系统间如何通信（事件/标记/直接调用）、有无统一动作执行与条件判断、生命周期如何约定、数据类型与接口定义分布在哪里等。所有结构概念（层名、系统名、接口名、文档中的术语与代码中的对应关系）均在本步探索中得出，技能正文不预先引用任何代码里的类、函数、接口或文件名。
- 目标：建立对分层边界、通信方式、依赖注入方式、数据驱动边界的完整理解，且该理解完全来自探索结果。

### 第二步：与架构设计文档核对并同步

- 在项目中定位架构设计文档（由第一步探索得到其位置）。
- 逐项核对：目录结构、系统列表、接口约定、事件清单、数据流、配置驱动清单（具体项名均以探索与文档为准）。
- **不一致时以代码为准**：文档中多出的（如文档有但代码没有的模块）、代码有但文档未写的（新模块、新 UI、新系统），都要在文档中增删改，使文档与当前代码一致。
- 同步完成后可简要列出「已同步项」供用户确认。

### 第三步：架构 Review 并列出问题

按下面「审查问题类型」逐类检查，只做发现与记录，不做修改。

**审查问题类型（需全部覆盖）：**

1. **耦合问题**
   - 同层系统是否直接持有或调用彼此（而非仅通过文档约定的「事件总线」「全局标记」等机制通信）。
   - UI 是否直接依赖多个业务系统实例，是否可收敛为通过事件/标记或少量稳定接口交互。
   - 核心层中「统一动作执行」与主入口的职责是否清晰；若某系统为执行动作而直接依赖另一系统且本可通过事件解耦，则记为耦合。

2. **过度设计**
   - 是否存在不必要的抽象层、多余接口或仅单实现的「框架式」扩展点。
   - 是否存在为「未来可能」而提前引入的复杂机制。
   - 是否存在重复能力（多套事件机制、多套状态存储等）。

3. **分层违反**
   - 下层是否依赖上层。
   - 渲染层是否依赖系统层或 UI 层（除文档允许的少量回调或解析接口外）。
   - 系统层是否直接操作底层渲染或 DOM，而非通过封装好的渲染/实体接口。

4. **生命周期与销毁**
   - 各系统与实体是否具备完整的销毁逻辑（事件监听、定时器、渲染对象、缓存引用是否在销毁时清理），销毁后是否仍有残留。
   - 重新初始化后行为是否与首次一致；场景或模块切换时旧资源与监听是否完全释放。

5. **数据驱动与配置驱动**
   - 文档规定「必须由配置驱动」的内容是否在代码中被硬编码（业务 ID、名称、文案等）。
   - 条件判断是否统一走文档约定的「唯一条件源」机制，是否存在散落的本地状态分支。

6. **统一动作执行**
   - 文档规定的「统一动作执行」场景（如遭遇结果、演出、对话触发的行为、任务奖励等）是否都经由该机制执行，是否存在系统内部直接改状态而未走统一入口的路径。

7. **接口与约定**
   - 文档约定应实现统一系统接口的模块是否都实现了约定生命周期（初始化、更新、序列化、反序列化、销毁）；上下文是否满足各系统实际需要，是否导致绕过注入的变通做法。

8. **其他**
   - 循环依赖。
   - 全局单例或跨模块直接引用实例（未通过构造函数或约定方式注入）。
   - 事件命名与文档不一致或缺失。
   - 职责混杂：同一模块既管状态又管 UI 又管输入等。

### 第四步：输出审查报告并归档，不主动修复

- 按问题类型汇总所有发现，每条注明：**类型、位置（由探索得到的模块/层级或文件）、简要描述、与哪条架构原则冲突**。
- 报告正文开头注明**审查时间戳**（建议格式：YYYY-MM-DD HH:mm）。
- 将报告保存到 **artifact/Reviews** 目录下，文件名建议包含日期便于历史存档（例如按日期或日期+时间命名）。
- 明确告知用户：「当前仅完成审查与报告，未做任何代码修改。若需要修复，请回复『开始修复』。」

### 第五步：仅在用户说「开始修复」后执行修复

当用户明确回复「开始修复」或等价表述时：

1. **列出修复计划**：按优先级或依赖关系排序的条目列表（每条对应一个具体修改点），便于逐项执行和复查。
2. **逐条修复**：每完成一条，做一次**局部复查**（检查该修改是否影响其他模块、是否引入新耦合或违反分层），确认无误后再进行下一条。
3. **全部完成后**：简要总结已修复项，并通知用户「修复已完成，请验证功能。」

## 报告格式建议

```markdown
## 架构审查报告

**审查时间戳：** YYYY-MM-DD HH:mm

### 文档同步说明
- （已与架构设计文档同步的变更列表）

### 问题列表

#### 1. 耦合
- [ ] 位置：（探索得出的模块/层级或文件）。描述。违反原则：……

#### 2. 过度设计
- （同上格式）

#### 3. 分层违反
- （同上格式）

...（其他类型）

### 下一步
当前仅做审查与报告，未修改代码。如需修复请回复「开始修复」。
```

报告需保存到 **artifact/Reviews** 目录，文件名含日期或时间戳以便历史存档。

## 注意事项

- 审查时以架构设计文档中的「架构原则（铁律）」和「配置驱动清单」为基准；文档与代码不一致时，**以代码为准**并同步文档，再基于同步后的文档做问题判断。
- 修复阶段禁止一次性大改多文件；每修一点就做一次影响面检查。
