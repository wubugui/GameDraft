# Feature Iteration

用于处理不改变玩法设计前提下的代码迭代、重构和实现细化。先确认不触碰玩法，再基于架构文档实施，最后做局部回归 review。

## 何时使用

当需求主要属于以下类型时使用本技能：

- 架构修复、解耦、生命周期修复
- 重构、代码整理、模块职责收口
- 性能优化、资源加载优化、事件链修复
- 存档链路修复
- UI 表现完善、交互细化、可读性提升
- 工具链、编辑器辅助、工程配置优化
- 不改变玩家规则、任务设计、资源流和玩法结果的功能迭代

## 必须遵循的流程

### 1. 先确认这不是玩法迭代

- 先快速判断本次改动是否会影响玩法规则、进度逻辑、对话结果、任务奖励、遭遇选项、资源消耗或玩家可见的行为规则。
- 如果会影响，停止使用本技能，转为 `gameplay-iteration`。

### 2. 先读架构文档和相关代码

- 先读取 `游戏架构设计文档.md` 的相关章节
- 再读取直接相关的代码文件
- 必要时读取最近的 review 文档，避免重复引入已知问题

只关注本次改动直接相关的模块，不做无边界扩散。

### 3. 以最小必要改动实现目标

- 优先延续现有模式
- 优先局部修复，不顺手大改
- 不为了“更优雅”而扩大改动范围
- 除非用户明确要求，否则不要改玩法文档

### 4. 实施时必须遵守架构铁律

对照 `游戏架构设计文档.md`，重点避免：

- UI 直接修改业务状态
- 同层 system 直接互相依赖
- 系统层直接操作不该直接操作的渲染细节
- 业务文案和内容定义硬编码
- 绕过 FlagStore、ActionExecutor、统一生命周期约定

### 5. 如果发现会影响玩法，立即停止并上报

若实施过程中发现当前技术改动实际上会改变：

- 玩家结果
- 任务条件
- 资源获取或消耗
- 规矩/遭遇/对话逻辑
- 文档定义的玩法行为

则不要继续按纯功能迭代处理。应明确说明影响点，并建议切换到 `gameplay-iteration`。

### 6. 完成后做局部回归 review

只 review 本次改动涉及的局部范围：

- 改动文件和直接关联文件
- 相关事件链、动作链、存档链
- 是否引入新的耦合、分层违反或生命周期残留
- 若改到 UI，确认窗口大小合适且所有内容都能被看到

### 7. 发现问题先修复，再结束

- 局部 review 发现问题时，先修复
- 修复后再次复查
- 只有局部范围稳定后才结束

### 8. 结束时的汇报格式

最终汇报应包含：

- 本次属于哪类非玩法迭代
- 主要修改了哪些模块
- 是否确认未改变玩法设计
- 做了哪些局部回归检查
- 是否发现并修复了问题
- 剩余风险或后续建议

## 额外限制

- 不要创建任何单独测试用例
- 不要把小修变成大重构
- 不要跳过局部回归 review
- 不要在“疑似影响玩法”时继续硬做
