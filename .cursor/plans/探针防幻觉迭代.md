## 迭代二：引用「严格校验」（用户已选定）

**核心：摘录必须校验**

- 对 `--- 引用 ---` 中 JSON **每一条**校验：
  - `path` 须在本轮对话里曾被 `read_file` 打开；
  - **`quote` 必须经过子串校验**（规范化空白后须为该 path 对应**同轮工具返回正文**的子串）。
- `[]`：仅当正文为「未找到/未记载」类**拒答**时通过；若正文含多条「灵感」却 `[]`，判失败（细则实现时可再收紧）。

**校验失败时：对用户仍严格（不展示带病稿）**

- **不向用户展示**未通过校验的那一版全文（避免假引用污染界面）。
- 可选：失败稿写入 trace / 日志供排查。

**校验失败时：若要再给 Agent 一次机会——禁止无脑重跑**

- **不得**复用与「未检测到 `read_file`」相同或类似的泛泛 nudge（例如只喊「请调用 read_file」）。
- **应**构造**针对性系统/用户消息**，至少包含：
  - **发生了什么**：例如「引用条目 #2：`chronicle/.../x.json` 的 `quote` 与本轮该文件 `read_file` 返回内容不匹配（或该 path 本轮未读取）」；可列出失败条目的 path、摘录前若干字。
  - **应该怎么做**：例如「请重新对该 path 执行 `read_file`，从返回正文中**逐字**截取 ≤40 字写入 `quote`；若仍无依据则删除该条灵感并改用『未记载』+ `[]`；不得编造摘录」。
- 修复轮次可限制为 **1 次**（或配置化）；若修复轮仍失败，再向用户返回**固定短说明**（引用未通过校验，请重试或换模型），**不再**无限重试。

**实现**：`probe_citation_verify.py`（解析 JSON、子串比对、生成上述「诊断 + 指令」文案）+ [`run_probe_user_turn`](f:/GameDraft/tools/chronicle_sim_v2/core/agents/probe_agent.py) 内编排：先校验 → 失败则带 history 追加诊断消息再 `run_agent_traced` → 再校验 → 仍失败则返回固定拒绝文。

**不采用**

- 仅追加「温和」脚注、仍展示未校验全文。
- 引用失败后与「未读文件」共用同一套笼统提示、无差别第二次 `run`。
