# 制作人协作法(先访谈对齐再出稿)

2026-07 位面/背尸设计长会话中经多次纠偏确立;制作人明确指定按
Anthropic《Finding Your Unknowns》的方式协作。

## 适用时机

制作人给方向但留空细节的系统性设计/规划任务(新机制、编辑器大改、内容框架)。
执行型小改不适用。

## 阶段骨架

1. **亮 unknowns**:列出"答案会改变架构"的未知项清单。完成判据:每项都说得清
   为什么不同答案会导向不同设计。
2. **访谈对齐**:一次一问,优先要参照物(他脑中已有的名字/游戏/系统)而非让他写描述。
   完成判据:能用制作人自己的词汇复述模型并被确认。
3. **出稿**:通俗短句写方案。完成判据:制作人能读懂并逐点表态。
4. **签字**:定稿后给可证伪表态("什么情况我会改主意")。完成判据:经得起
   "你确定吗"的校准追问。

## 判断点

- 制作人**顿悟式命名**(如"位面""像 cutscene 一样全局注册")是最高价值信号——
  立即抓住展开,放弃自己原有框架。
- 他嫌方案难懂→切"执行策划向制作人汇报"的大白话重讲,不是补更多细节。
- 被质疑"不系统/不可维护"→回到骨架重来,不打补丁;结构与可迭代性优先级常高于进度。

## 分工契约

agent 出结构与选项、扛一致性、敢签字担责;制作人拍模型与取舍;
他要"帮我想、帮我理清楚"式伙伴,不是执行器。

## 已知死路

- 拿行业最佳实践(槽位/零件/Lyra 式)填制作人留白——填的是自己的地图不是他的疆域,必返工。
- 让制作人写长描述代替给参照物——信息密度低且失真。
- 快拼/裸 flag/"实现就了事"式提案——被斥为不可维护,整轮作废。

## 向下指针

- 玩法设计铁律(机制必须高频占用玩家注意力)→ content 域 norms;
- 案例:位面系统的取舍史 → runtime 域 decisions(位面)。
