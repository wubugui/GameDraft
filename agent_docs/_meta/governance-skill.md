# agent_docs 一键治理(治理流程权威文件)

> 本文件是治理流程的**唯一权威源**,客户端无关——任何 agent(Claude、Codex、Cursor…)
> 执行治理都照本文件做。发现方式 = 治理台 CLI(`python3 agent_docs/_meta/cli.py`);
> 各客户端只装一个 CLI 薄壳(`agent-docs-cli`),任何壳不得复制本正文。
>
> **触发时机**:用户说"治理 agent 文档、一键治理、更新知识库、建库、蒸馏记忆、
> govern agent docs"。首次运行即建库 run(见末节)。
> 日常**单件**收编(零散方法论/经验/坑想入库)不走本流程,走同级的
> [intake-skill.md](intake-skill.md);本流程周期性对账时把 intake 收编记录当证据回看。

对 `agent_docs/` 做一轮完整治理。**开工前必读三件**(硬前置,不读不许动):
`agent_docs/_meta/constitution.md`(判据)、`agent_docs/_meta/schema.md`(格式契约)、
`agent_docs/_meta/governance-log.md`(已拍板决策,禁止重新问人)。

## 总原则

- **只提供元方法**(宪法 §0):本流程与一切治理流程都按"阶段=目的+完成判据+结构保证"
  写作,不铺操作步骤;正确性靠结构(盲重建/对抗校验/审批面/机械门禁)兜底,
  "怎么做"由执行 agent 现场智能补齐。
- **报告是给 agent 读的,人只收决策题**:题面=一行题干+一行证据+一行后果;
  没有审批级发现就零提问,静默完成后汇报摘要。
- **成本形状**:机械体检分钟级、可常跑;深度治理(盲重建)大 token、按需跑
  (大迭代前、大合并后、用户点名)。运行前告知用户本轮预计范围。
- 治理只写 `agent_docs/**`;库外仅允许"旧址立牌"(被迁移文档原地留一行指针)。
  **CLAUDE.md 一个字都不碰**(接线未获批准,见 governance-log)。

## 流程

### 0. 机械体检

```bash
python3 agent_docs/_meta/audit.py
```

error 先修(锚点失配、格式违规、索引重生成)。这层不需要判断力,先清场。

### 1. 定界

和 governance-log 对表:哪些域该做深度治理(用户点名 > 偏差积压多 > 距上次治理久)。
每域独立可治,不必全库一次做完。

### 2. 证据管线 A:代码锚定(norm 事实部分 / mechanism / recipe)

按域分片派**盲重建** agent(建议并行子 agent;规模大时经用户同意用 workflow):

- 输入:只给代码与数据文件的入口,**禁止读 agent_docs 旧文档**——先读旧档再看代码会
  确认偏误,漂移就查不出来了。
- 产出:该域机制图谱(机制→权威源→硬契约→坑),结构化返回。
- 对账:治理主脑拿图谱逐条比对库内声明,判**属实 / 漂移 / 过时 / 缺失**。

### 3. 证据管线 B:实践锚定(method / decision)

证据源:`_meta/inbox/` 偏差记录、governance-log 里的 intake 收编记录、`artifact/` 实操产物、
git log、(可用时)会话转录。分诊三态:

- 高频使用 + 零偏差 → **静默确认**:`last_governed` 盖章续期;
- 偏差积压 → **蒸馏改稿**:多条偏差的共性=骨架问题(升审批),孤例=判断点备注(自动落地);
- 久未使用(audit 的 info 信号)→ **标疑**:status 改 suspect + 出"还留吗"审批题。

蒸馏改稿与重审 method 时按 **method 三原则**(constitution §1:疆域不地图/正交可复用/
组合不强耦合)裁剪:把写死的步骤降为可选提示,把强耦合方法拆成正交 method + 组合层。

### 4. 自动落地(不问人)

指针/锚点刷新;新增已知坑/死路(带证据引用);补缺失机制卡;标记过时;
按删除测试/越权测试裁剪措辞;蒸馏完的 inbox 记录删除;重跑 audit 重生成索引。
**记忆升格**(政策已批):蒸馏中发现 agent 私有记忆里的项目契约类内容,升格入库、
原记忆改留指针;个人偏好类留在记忆,不动。

### 5. 审批面(仅四类,AskUserQuestion 选择题)

①不变量增/删/改;②method 骨架变更(增删阶段/改分工契约);③整篇删除或废弃;
④与既有 decision 冲突的发现。每题一行题干+一行证据+一行后果。四类之外不许上升。

### 6. 收尾

```bash
python3 agent_docs/_meta/audit.py          # 必须零 error
```

追加 `_meta/governance-log.md`:改了什么、人拍板了什么、蒸馏掉哪些 inbox、下轮建议。
向用户汇报一段摘要(改动数量级+拍板结果),不甩全文报告。

## 首轮建库 run 的特别流程

首轮没有旧库可对账,流程改为"蒸馏建库":

1. 迁移 `docs/运行时开发规范.md` → `agent_docs/runtime/norms.md`(内容已经用户审批,
   补 frontmatter 即可;旧址立牌)。
2. 逐域蒸馏素材:agent 私有记忆(约七成是项目契约,升格政策已批)、`docs/` 混堆、
   `artifact/Reviews|Plans`(只蒸馏不治理)、`.cursor/skills/` 内嵌的知识段(迁出成卡,
   skill 原文改为引用;skill 的可执行流程部分留在原处)。
3. 每篇入库都过收录六条 + 写作高度两测试(constitution §1 §3);拿不准收不收→按类型:
   规则类升审批,事实类不收(留在代码里)。
4. 各域 norms.md 必须写入**偏差记录义务**条款(constitution §4)。
5. 首轮的审批题预期集中在:各域 norm 的定稿、旧文档整篇废弃清单。

## 偏差记录义务(供其它 skill / agent 引用)

工作中发现现实与库内文档打架:收尾时向 `agent_docs/_meta/inbox/` 丢一条记录
(格式见该目录 README;三行,零门槛,不需审批)。这是 method 类知识唯一的保鲜来源。
